# 社区公开资料一致性与互动冷启动设计

## 1. 背景

初始内容包已经提供 8 个社区账号和 40 篇文章，但公开作者资料在不同读取链路中没有形成统一契约：

- 文章详情只返回作者 ID、昵称和头像，没有返回 handle 与简介，导致作者卡显示占位文案，并退回到按 ownerId 筛选文章的链接。
- Feed 的前端类型声明了 authorId 与 authorHandle，后端响应实际没有这些字段，文章卡无法稳定跳转到标准个人主页。
- 评论查询已经联表读出作者 handle 和头像，但响应丢弃 handle，前端也没有展示头像。
- 用户更新昵称、头像、简介、handle 或兴趣标签后，文章详情缓存、Feed 缓存与 Elasticsearch 中嵌入的作者快照可能长期不一致。
- 8 个内容包账号中已有 6 个简介、2 个简介为空，所有账号都保存了兴趣标签，但公开个人主页没有返回或展示标签。
- 当前互动清单为空，44 篇公开文章缺少评论、关注、点赞、收藏与自然浏览分布。

本设计先建立统一、稳健的公开作者资料读取链路，再在该链路之上导入可审核、可重复执行、可回滚的社区互动包。

## 2. 目标

1. 以 MySQL 的 users 表作为公开作者资料的唯一事实来源。
2. 统一文章详情、Feed、搜索结果、评论与个人主页的作者身份字段。
3. 资料更新后，公开页面立即显示最新资料；缓存或 Elasticsearch 暂时不可用时仍可安全降级。
4. 为 8 个内容包账号补齐自然的个人主页资料。
5. 为现有公开文章增加克制、分散、符合账号兴趣的社区互动。
6. 互动内容必须先生成审阅页，用户验收后才允许正式写入数据库。
7. 所有正式导入必须先备份，支持幂等重跑，并且只管理清单明确拥有的数据。

## 3. 非目标

- 不公开性别、生日、学校、电话或邮箱；这些字段在缺少隐私开关前保持私有。
- 不替 rekyrice 账号自动评论、点赞、收藏或关注。
- 不批量生成珂朵莉评论。
- 不新增文章，不重写已验收的 40 篇内容。
- 不把 Seed 系统扩展为生产环境的持续自动运营系统。
- 不修改或收口其他 worktree。
- 不为旧 handle 增加历史别名与重定向；当前链接始终使用最新 handle。

## 4. 统一公开作者投影

### 4.1 数据模型

新增内部只读模型 PublicAuthorSnapshot：

- id
- handle
- nickname
- avatar
- bio
- tags
- createdAt

UserMapper 提供单用户读取与按 ID 集合批量读取。标签在 Mapper 或专用转换器中安全解析为不可变列表；非法 JSON 记录告警并降级为空列表，不让页面请求失败。

PublicAuthorQueryService 是公开作者资料的唯一应用层入口。首版不为该投影增加新的 Redis 或本地缓存：文章详情每次只多一次单行查询，Feed 每页只多一次 IN 批量查询，换取提交后立即可见和更小的失效面。

### 4.2 API 契约

保持已有字段兼容，只追加字段：

- FeedItemResponse：追加 authorId、authorHandle。
- PostDetailResponse：追加 authorHandle、authorBio；保留 authorTagJson 以兼容现有消费者。
- CommentResponse：追加 authorHandle。
- PublicUserResponse：追加 tags 列表。

前端 TypeScript 类型与后端记录保持逐字段一致。所有用户 ID 继续以字符串返回，避免 JavaScript 大整数精度问题。

### 4.3 响应时补全

文章与搜索缓存仍可保存昵称、头像等快照，作为数据库读取异常时的降级数据，但正常响应必须按 authorId 使用 PublicAuthorQueryService 覆盖为最新资料：

1. 文章详情从缓存或数据库取得文章主体和 authorId。
2. enrich 阶段读取当前 PublicAuthorSnapshot。
3. 用当前 handle、昵称、头像和简介构造最终响应。

Feed 的数据库、Redis 与 Elasticsearch 路径统一执行批量补全：

1. 每个原始 Feed 项必须携带 authorId。
2. 收集当前页唯一作者 ID。
3. 一次批量查询公开作者资料。
4. 按 ID 覆盖作者 handle、昵称和头像。

这一步放在用户态 liked/faved 补全之后、Controller 返回之前，确保公共 Feed、关注 Feed、个人文章 Feed、搜索结果和推荐列表采用相同规则。

评论继续使用现有联表查询以避免额外查询，但 DTO 必须保留已经查询到的 authorHandle，并在前端展示头像与个人主页链接。

### 4.4 降级行为

- 作者批量查询整体失败：记录结构化告警，保留缓存或 ES 中的作者快照。
- 单个 authorId 缺失：使用已有快照；快照也缺失时显示“已注销用户”，不提供个人主页链接。
- 简介或标签为空：仅相应区域显示克制的空状态，不影响文章与评论。
- 任何作者资料补全失败都不能让整页 Feed 或文章详情返回 5xx。

## 5. 缓存与搜索收敛

### 5.1 详情与 Feed 缓存

- 升级文章详情缓存布局版本，旧 JSON 不与新契约混用。
- Feed 缓存必须包含 authorId，但其昵称、头像与 handle 只作为降级快照。
- 资料更新不需要扫描并删除所有文章缓存，因为最终响应会由 MySQL 覆盖作者字段。
- 文章自身修改仍沿用 PostCacheInvalidator，不改变现有文章缓存失效语义。

### 5.2 Elasticsearch

搜索文档增加 author_handle，并确保 author_id 始终存在。现有索引采用可重复执行的增量 mapping 更新，不要求为新增字段破坏性重建整个索引。

资料更新事务同时写入持久化 Outbox 事件 AUTHOR_PROFILE_CHANGED，payload 只包含 eventId、userId 与 updatedAt。消费者：

1. 使用 eventId 做幂等保护。
2. 查询该用户所有已发布文章 ID。
3. 逐篇调用现有 SearchIndexService 重建文档。
4. 失败时进入已有重试与死信链路。

页面显示正确性不依赖该异步链路；Outbox 负责让“按作者昵称/handle 搜索”等索引行为最终收敛。Kafka、Canal 或 ES 暂时不可用时，资料更新事务仍然成功。

## 6. 前端公开资料体验

### 6.1 文章与 Feed

- AuthorCard 接收并展示 authorBio，作者名称与“查看作者所有文章”都链接到 /user/{handle}。
- ArticleReadingSidebar 使用 handle 跳转个人主页，不再退回 ownerId Feed。
- PostCard、推荐列表和侧栏作者入口统一使用 authorHandle。

### 6.2 评论

- 普通评论显示 32px 作者头像；无头像时显示昵称首字符。
- 作者昵称链接到 /user/{handle}。
- 珂朵莉评论继续使用角色展示，不伪装成普通用户。
- 回复层级保持一层，不改变现有评论树结构。

### 6.3 个人主页

公开显示：

- 头像、昵称、handle
- 1 至 2 句短简介
- 2 至 4 个兴趣标签
- 加入时间
- 文章数、关注数、粉丝数
- 文章列表

不显示性别、生日、学校、电话与邮箱。

## 7. 内容包账号资料

8 个账号全部补齐非空简介，每个简介保持 20 至 60 个中文字符，使用普通社区用户语气，不写完整的“人物设定卡”。原有兴趣标签继续保留并公开展示。

内容包账号增加 joinedAt，由清单管理其创建时间，分散在 2026 年 2 月至 6 月。只允许更新内容包身份映射明确拥有的账号，不修改站长账号或自然注册用户。

资料更新保持幂等：相同清单重跑不会继续改变 created_at、updated_at 或产生新身份。

## 8. 社区互动包

### 8.1 参与者与覆盖范围

- 发言和反应主体仅为 8 个内容包子账号。
- 可以互动 40 篇内容包文章与 rekyrice 的公开文章。
- 不生成 rekyrice 账号行为。
- 不生成珂朵莉批量评论。
- 不要求每篇文章都有评论或点赞，保留自然冷清区。

### 8.2 目标密度

第一版采用固定、可审阅的规模：

- 36 条评论：28 条顶级评论、8 条一层回复。
- 88 条反应：62 个点赞、26 个收藏。
- 14 条有向关注关系。
- 44 条最低浏览量声明，范围约 18 至 180；旧文章整体高于新文章，但保留离散波动。

评论覆盖约 22 至 26 篇文章；反应覆盖约 30 至 34 篇文章。rekyrice 的公开文章至少有 3 篇获得子账号互动，但不强制每篇覆盖。

### 8.3 内容规则

- 账号只评论其兴趣范围内、确实能读懂的文章。
- 顶级评论以一句具体观察、一个细节或一条个人经验为主，长度主要为 18 至 90 个中文字符。
- 回复必须回应父评论中的具体内容，不能只写“同意”“确实”“学到了”。
- 技术评论允许追问、补充边界或报告类似问题，不虚构不存在的生产经历。
- 动漫、游戏与生活评论避免重复文章摘要，不使用统一的文学腔。
- 不出现通用 AI 助手话术、总结式结尾、批量感叹号或整齐的三段论。

### 8.4 时间规则

- 评论时间必须晚于目标文章发布时间。
- 回复时间必须晚于父评论。
- 评论、关注与账号加入时间不得晚于正式导入时间。
- 同一账号的行为分散到不同日期和时段，不产生同一分钟内的大批写入痕迹。

### 8.5 外部文章引用

现有 interaction 定义只支持 postSeedKey，无法引用内容包之外的 rekyrice 文章。扩展为二选一引用：

- postSeedKey：引用内容包拥有的文章。
- postSlug：引用现有站长公开文章。

Validator 要求二者恰好存在一个。postSlug 只允许匹配 site.ownerUserId 所属、published 且 public 的文章；找不到、重复或归属不符时在事务开始前失败。清单不修改这些外部文章，只把它们作为互动目标。

## 9. 幂等、所有权与回滚

- 每条评论、关注和反应都有稳定 seedKey，并写入 Seed identity。
- 重跑相同版本只校正清单拥有的互动，不重复增加计数。
- 删除清单中的评论或关注时，只停用同 namespace 拥有的记录。
- 点赞和收藏只操作内容包账号对应的位图成员，不覆盖真实用户状态。
- views 使用 minimumCount 语义，只补到最低值，不降低真实浏览量。
- 外部站长文章只允许增加或撤销清单拥有的互动，禁止修改文章本身。
- 数据库写入与 identity 更新在同一事务内；Redis 计数、缓存和 ES 在提交后收敛。
- 提交后阶段失败返回 partial 报告，下一次相同导入必须可继续收敛，不能重复累计。

## 10. 审阅与正式导入

生成项目内、Git 忽略的静态审阅页：

- 路径：.codex-tmp/community-interactions/review/index.html。
- 评论区逐条展示作者头像、昵称、目标文章、正文、父回复和时间。
- 统计区展示每个账号的评论数、点赞数、收藏数、关注关系与兴趣匹配情况。
- 文章分布区展示哪些文章无互动、哪些文章互动偏高。
- 审阅页不包含密码、Token、邮箱或数据库连接信息。

正式导入门禁：

1. 内容包结构、引用、时间与质量校验全部通过。
2. 用户验收审阅页。
3. 使用现有 backup-mysql.ps1 生成项目内 Git 忽略备份。
4. dry-run 报告通过。
5. 正式导入。
6. 核对 MySQL、Redis 计数与 Elasticsearch 文档。
7. 浏览器验收 Feed、文章详情、评论和个人主页。

备份与审阅文件不提交 Git；accounts.yml、interactions.yml、导入代码、测试与设计文档提交。

## 11. 测试策略

### 11.1 后端单元测试

- PublicAuthorQueryService 单个与批量读取、标签容错和缺失用户降级。
- FeedItemResponse 的 DB 与 ES 映射都保留 authorId，并被当前资料覆盖。
- PostDetailResponse 缓存命中后仍返回最新简介、handle、昵称和头像。
- CommentResponse 保留 authorHandle。
- ProfileService 更新公开字段时写入一次 Outbox；私有字段更新不触发不必要重建。
- Outbox 消费幂等、失败重试，并只重建该作者的已发布文章。
- ContentPackValidator 校验外部 slug 归属、二选一引用、时间顺序、重复互动和自关注。
- 内容包重跑不会重复评论、计数或关注关系。

### 11.2 集成测试

使用 Testcontainers 覆盖：

1. 创建用户与文章并预热详情、Feed 缓存和 ES。
2. 更新昵称、头像、简介、handle 与标签。
3. 验证详情、Feed、评论和个人主页立即读取新资料。
4. 消费 Outbox 后验证搜索文档收敛。
5. 模拟 ES 或 Kafka 不可用，确认资料更新成功、页面从 MySQL 正确显示，恢复后可重试。

### 11.3 前端测试

- AuthorCard 展示真实简介并使用 canonical handle 链接。
- ArticleReadingSidebar、PostCard 与评论作者链接一致。
- 评论头像、空头像和珂朵莉分支。
- 个人主页展示兴趣标签且不渲染私有字段。
- API 类型契约测试防止后端字段再次遗漏。

### 11.4 完整验证

- 后端定向测试与 mvn test。
- Testcontainers 集成测试。
- 前端 npm run test:run 与 npm run build。
- 内容包 validator、dry-run 与重复导入测试。
- Playwright 验证桌面与移动端的个人主页、文章作者卡和评论区。

## 12. 实施顺序

1. 用失败测试固定现有资料契约缺口。
2. 实现 PublicAuthorSnapshot 与批量读取。
3. 统一详情、Feed、搜索和评论 DTO。
4. 增加资料更新 Outbox 与搜索收敛消费者。
5. 修复前端作者入口、评论头像和个人主页标签。
6. 扩展内容包账号 joinedAt 与外部文章引用。
7. 编写账号资料和互动清单。
8. 生成审阅页并等待验收。
9. 备份、dry-run、正式导入和运行时核对。
10. 完整测试、提交、推送并创建 PR。

## 13. 验收标准

- rekyrice 已保存的简介能在其文章作者卡中显示。
- Feed、文章详情、阅读侧栏和评论均能进入正确个人主页。
- 资料修改后，不刷新文章缓存或重建 ES 也能在页面看到最新资料。
- ES 恢复后能够通过 Outbox 自动收敛作者资料。
- 8 个内容包账号都有自然的公开简介、兴趣标签与分散加入时间。
- 互动数量符合清单，重跑不会重复累计。
- 没有任何系统生成的 rekyrice 行为或批量珂朵莉评论。
- 正式数据库导入前存在可恢复备份。
- 所有自动化测试、生产构建与浏览器验收通过。
