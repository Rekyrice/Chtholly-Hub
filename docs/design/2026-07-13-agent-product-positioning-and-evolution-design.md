# Chtholly Hub Agent 产品定位与演进设计

## 1. 文档目的

本文定义 Chtholly Hub 中 Agent 的产品角色、能力边界、技术映射与演进条件，回答以下问题：

- 为什么内容社区需要 Agent，而不只是一个聊天框？
- 读者协作、创作协作和后台智能任务之间是什么关系？
- 为什么当前采用单 Agent，而不是按能力名称拆成多个 Agent？
- 哪些能力已经进入主链，哪些仍是条件能力或演进方向？
- 当系统未来需要多 Agent 时，应由哪些可验证条件触发？

本文是产品与架构之间的决策文档，不替代实现说明。当前代码主链、Contributor、扩展开关、Memory、Trace 与测试入口以 [Agent 系统](../architecture/agent-system.md) 为准；跨组件时序以[核心请求链路](../architecture/request-flows.md#8-agent-websocket上下文工具与记忆)为准。

## 2. 背景与核心矛盾

Chtholly Hub 从个人动漫博客演进为内容社区后，用户面对的不再只是静态文章，还包括站内内容、作品与角色资料、关系状态、互动历史和持续增长的知识。传统博客搜索只能返回页面，通用聊天模型又不了解当前页面、站内事实和用户关系，容易产生三类割裂：

1. **内容与问答割裂**：用户阅读文章时产生问题，却需要离开当前页面重新搜索。
2. **生成与事实割裂**：模型能够生成自然语言，但不知道哪些信息来自站内内容、哪些来自外部资料，也缺少受控工具。
3. **一次回答与持续协作割裂**：聊天可以给出一句建议，却不能稳定管理上下文、执行步骤、失败恢复和质量验证。

因此，项目需要的不是“给博客接一个大模型”，而是一个位于内容、用户和工具之间的受约束协作层：它理解用户所处场景，只调用被允许的能力，以站内证据和领域数据完成任务，并让执行过程可以追踪、降级和评估。

## 3. 一句话定位

**Chtholly Hub Agent 是内容社区内的情境协作者：当前以页面、站内知识和用户关系为上下文，通过有界推理与受控工具帮助用户理解内容；后续在安全任务状态与审批机制完善后扩展到创作协作，并让相同的工具、Trace 和评测合同服务于可审计的后台智能任务。**

这一定义包含四个不可省略的约束：

- **属于内容社区**：Agent 的价值来自站内内容与真实用户场景，不是脱离产品的通用聊天入口。
- **理解情境**：页面、身份、关系、历史和知识必须以明确顺序进入上下文。
- **受约束执行**：工具、步数、时间、权限和副作用必须由系统控制。
- **可验证**：回答与工具执行需要 Trace、测试和评估入口，而不是只凭主观体验判断质量。

## 4. 三层能力模型

三层模型用于组织产品能力，不代表每个请求都会从上到下依次升级，也不代表三个独立 Agent。读者与创作协作体现用户任务深度；后台智能任务主要体现正交的执行生命周期。三者复用部分 Context/Tool 合同、权限原则、Trace 与评测基础，但不共享同一份上下文、记忆或运行状态。

### 4.1 第一层：读者协作

读者协作解决“用户正在消费内容，如何在原场景内继续理解”的问题，是当前 Agent 的核心定位。

典型任务：

- 解释当前文章中的概念、人物、作品和关系；
- 检索站内相关文章与全文结果；
- 查询 Bangumi 作品、角色和人物资料；
- 结合当前页面与会话历史回答追问；
- 在证据不足或依赖不可用时明确降级，而不是补全不存在的事实。

产品价值是减少阅读中断，让静态内容变成可追问、可关联的知识入口。技术重点是页面上下文、检索质量、只读工具、流式交互、引用与事实约束。

当前状态：**已进入交互主链，但检索评测、引用绑定、并发 turn 隔离和最终回答上下文仍需继续完善。**

### 4.2 第二层：创作协作

创作协作解决“用户要在资料和规则约束下完成多步骤内容任务”的问题，是读者协作的自然延伸，而不是把 Agent 改造成通用 Code Agent。

典型任务：

- 根据选题检索站内与领域资料；
- 在用户给定语气、长度、结构和事实范围内生成大纲；
- 对草稿做事实核对、结构检查和修改建议；
- 将任务拆成检索、整理、生成、审校等有界步骤；
- 在执行写入或发布前请求用户确认。

创作协作的核心不是“模型会写文章”，而是**在明确状态、证据和约束下执行任务**。它要求比读者问答更严格的任务状态、草稿版本、来源引用、人工确认、幂等写入和回滚机制。

当前状态：**属于下一阶段设计。现有运行时和只读工具提供基础，但安全写工具、任务状态机、审批与版本恢复尚未进入主链。**

### 4.3 第三层：后台智能任务

后台智能任务解决“无需用户保持连接的周期性或事件驱动工作”，包括：

- 内容索引与知识抽取；
- 质量检查与失败样本归类；
- 主题聚类、关系补全和内容治理建议；
- 定期离线评测与回归比较；
- 可恢复的长时资料整理任务。

这一层与前两层的差别不是有没有大模型，而是生命周期：任务需要排队、持久状态、重试、幂等、资源配额、人工审计和结果回放。同步 WebSocket 循环不能直接承担长时后台工作。

当前状态：**仓库已有索引、Trace、质量评估、知识图谱和部分后台扩展能力，但尚未形成统一的持久任务编排平台。**

后台 Worker 的默认定位是普通持久任务执行器：它按确定的输入、状态机与重试策略运行，不因调用模型就成为自治 Agent。只有任务具有独立目标、决策和权限边界时，才进入第 13 章的多 Agent 评估。

### 4.4 两类协作与后台执行平面

```text
用户任务深度：读者协作 ──扩展──▶ 创作协作
              页面问答           有约束的多步骤内容任务

执行生命周期：同步单轮/短时 ─────▶ 持久后台执行平面
                                排队、重试、审计与回放
```

三类能力对基础合同的需求不同：

| 能力 | 读者协作 | 创作协作 | 后台智能任务 |
|---|---|---|---|
| 页面与用户上下文 | 必需 | 必需 | 按任务需要 |
| 站内/外部查询工具 | 只读 | 只读 + 受控写入 | 批量、可恢复 |
| 执行时长 | 单轮、短时 | 多步骤、中等 | 长时、异步 |
| 状态 | 会话历史 | 任务与草稿版本 | 持久任务状态机 |
| 人工确认 | 通常不需要 | 写入前必需 | 高风险结果发布前必需 |
| 失败处理 | 本轮降级 | 恢复步骤或回滚版本 | 重试、补偿、死信与重放 |
| 评价 | 回答质量 | 任务完成度与事实性 | 吞吐、成本、恢复与质量趋势 |

## 5. 典型用户旅程

### 5.1 阅读中的即时追问

用户在文章详情页询问：“文中提到的角色还出现在哪些作品里？”

1. WebSocket 入口绑定用户与会话，接收当前页面信息。
2. Context Engine 按身份、关系、页面、知识、工具、历史和问题顺序构造上下文。
3. 运行时判断是否需要站内检索或 Bangumi 工具。
4. 工具参数经过校验并限时执行，结果作为 observation 回到有界循环。
5. 最终答案流式返回，执行步骤进入 Trace，会话历史写入 Memory。

这个旅程验证第一层能力：用户不离开阅读现场，Agent 又不依赖模型记忆臆测站内事实。

### 5.2 约束下的内容创作

用户要求：“根据这些资料生成一份不剧透、800 字以内、包含来源的文章大纲。”

目标流程：

1. 创建带约束、资料范围和输出格式的创作任务。
2. 先检索与归档证据，再生成结构，不直接写入正式内容。
3. 对事实、剧透范围、长度和来源做独立检查。
4. 向用户展示草稿、证据与修改差异。
5. 用户确认后才写入草稿区；发布仍走现有权限与内容流程。

这里借鉴 Code Agent 的计划、执行、验证和审批机制，但任务对象是内容与知识，不是代码仓库。

### 5.3 无连接的后台质量任务

系统定期对近期 Agent Trace 和内容变更进行分析：

1. 调度器创建持久任务并记录输入快照。
2. Worker 执行索引、质量评估或失败模式归类。
3. 每一步记录状态、耗时、模型/工具版本与错误。
4. 可重试错误按策略恢复；不可恢复结果进入人工检查队列。
5. 结果只有通过阈值或人工确认后才影响正式索引或内容状态。

这个旅程验证第三层能力：连接只是观察渠道，任务本身不能依赖浏览器持续在线。

## 6. 产品边界

### 6.1 不是普通聊天框

普通聊天框以自然语言生成作为终点；Chtholly Hub Agent 以“在社区情境中完成可验证任务”为目标。页面上下文、工具权限、证据来源、终止条件和失败行为属于产品能力，不是 Prompt 的附属细节。

### 6.2 不是通用 Code Agent

Code Agent 的主要对象是代码、终端、文件和开发流程，强调修改、测试和提交。Chtholly Hub Agent 的对象是文章、作品资料、社区关系和创作状态。两者可以共享计划—执行—观察—验证的运行时思想，但不能直接复用权限模型：内容写入、发布、引用和用户隐私需要独立规则。

### 6.3 不是独立研究平台

研究型能力可以作为后台扩展，用于知识抽取、聚类和评测；当前产品不以通用科研工作台为核心。只有当研究任务形成明确用户、输入输出、资源预算和评价标准时，才进入主产品路线。

### 6.4 不是无边界人格陪伴

角色表达增强连续体验，但不能覆盖事实性、权限和安全约束。角色状态、情绪与主动行为属于可选扩展；关闭这些扩展后，Core 仍应完成页面理解、工具执行和可追踪回答。

## 7. 为什么当前采用单 Agent

### 7.1 决策

当前使用一个面向用户的 Agent Core，通过 Contributor、Tool、扩展开关和任务类型承载不同能力，不按“读者、创作、后台”直接拆成三个自治 Agent。

### 7.2 三层能力不等于三个 Agent

产品能力名称描述用户目标；Agent 边界描述自治执行、状态所有权和故障隔离。读者协作与创作协作会共享身份、页面、知识、工具、记忆和权限，只是任务深度不同。此时拆成多个 Agent 会引入新的问题：

- 谁判断请求属于哪个 Agent；
- 多个 Agent 如何共享和合并上下文；
- 工具权限和写入确认由谁负责；
- 循环互相调用时如何终止；
- Trace、成本和失败归因如何贯通；
- 同一事实在多个记忆中冲突时谁是权威。

在缺少稳定自治边界前，多 Agent 只会把模块化问题变成分布式协调问题。

### 7.3 当前单 Agent 的隔离机制

当前架构不是巨型 Service，而是多个明确边界：

- `ChthollyAgent`：一次用户 turn 的编排边界；
- `ContextEngine`：按稳定顺序组合上下文；
- `AgentLoopExecutor`：执行有最大步数的决策循环；
- `AgentLlmInvoker`：统一模型调用与流式入口；
- `AgentToolExecutor`：参数校验、用户上下文、超时和结果归一化；
- `AgentTool`：工具合同；
- `AgentMemoryStore`：会话记忆；
- Trace/Observation：执行记录和观测；
- 扩展组：在不反向污染 Core 的前提下加入领域能力。

这组边界先解决可理解性、测试性和降级问题，同时保留未来拆分的接口。

## 8. 产品能力到技术架构的映射

| 产品需要 | 当前技术边界 | 设计意义 |
|---|---|---|
| 知道用户正在看什么 | `PageContextContributor` | 把页面情境显式注入，而不是让模型猜测 |
| 维持角色与用户连续关系 | Identity/Relationship/History + Memory | 区分稳定身份、关系状态与短期对话 |
| 基于站内证据回答 | Knowledge Contributor、RAG、全文工具 | 将内容检索与自然语言生成解耦 |
| 查询领域资料 | Bangumi 工具 | 外部事实通过受控工具进入，不直接开放任意网络访问 |
| 执行多步骤任务 | 有界 Think-Act-Observe loop | 最大步数、未知工具和解析错误都有终止语义 |
| 防止任意调用 | Tool Registry + 参数 schema + 服务端用户上下文 | 模型只能选择系统注册的能力 |
| 流式展示过程 | WebSocket 事件协议 | 区分 think、act、observe、delta、final 与 error |
| 在依赖失败时保住主站 | `LLM_ENABLED` 与扩展开关 | AI 不是博客/社区启动前提 |
| 发现质量和可靠性问题 | Trace、Observation、离线评测 | 从“感觉有效”转向可回归、可归因 |

### 8.1 Context 是产品状态平面

Context Contributor 不是单纯拆 Prompt 字符串，而是定义哪些产品状态可以进入一次执行、由谁提供、顺序如何以及失败是否允许降级。Core 始终保留 Identity、Relationship、Page、Knowledge、Procedural、Tools、History 和 Question；Mood、Graph 等扩展插入固定位置。

创作协作进入主链前，应增加任务约束、草稿版本和审批状态等独立 Contributor，而不是把所有内容追加到 History。

### 8.2 Tool 是能力与权限平面

工具将模型意图转换为服务端可校验调用。当前以只读检索工具为主，适合读者协作。未来写工具必须额外具备：

- 明确的用户权限与资源范围；
- dry-run 或变更预览；
- 用户确认；
- idempotency key；
- 审计记录；
- 失败补偿与版本回滚。

写工具不能仅因为实现了 `AgentTool` 接口就自动获得执行资格。

### 8.3 Runtime 是受约束执行平面

有界 ReAct loop 负责决定是否调用工具，但不应承载长期任务调度。同步 turn 需要最大步数、模型/工具超时、总 deadline 和取消语义；后台任务需要持久队列、状态机与 Worker，应复用工具和 Trace 合同，而不是复用 WebSocket 生命周期。

### 8.4 Memory 是分层状态，不是无限聊天记录

当前 Redis List + Caffeine 主要承担短期会话历史。用户身份、关系、程序性规则、体验和知识图谱属于不同状态域，不能全部压入一个聊天摘要。未来长期记忆必须定义来源、更新条件、过期、纠错、用户可见性和删除机制。

### 8.5 Retrieval 是证据入口，不是质量结论

当前文章问答、站内语义检索、Elasticsearch 关键词检索和 Bangumi 实体查询提供多种证据来源。检索链路存在即代表“可以取证”，不代表相关性已经得到证明。需要以固定数据集评估 Recall@K、MRR/nDCG、引用正确率、无答案拒答和延迟。

### 8.6 Trace 与 Evaluation 是演进前提

没有 Trace，就无法区分模型错误、工具错误、检索错误和上下文错误；没有离线评测，就无法判断增加记忆、工具或多 Agent 是否真的提高任务成功率。Trace 和评测不是附属后台，而是所有后续复杂化的准入条件。

## 9. Core 与扩展边界

### 9.1 Core 必须保持的能力

Core 负责：

- 交互入口与用户绑定；
- 一次 turn 的编排；
- Context 与 Tool 合同；
- 有界执行、超时和终止状态；
- 会话记忆；
- Trace 与基础指标；
- 关闭扩展后仍可工作的中立实现。

### 9.2 扩展适合承载的能力

Mood、Graph、Learning、Experience、Proactive、Content 和 Comment 等扩展负责变化频率更高、并非每次对话必需的领域行为。扩展可以提供 Contributor、服务或后台任务，但不得要求 Core 反向依赖某个具体实现。

### 9.3 扩展不是自治 Agent

扩展组表达编译/装配与领域隔离，不代表它拥有独立目标、循环、记忆和权限。具有持久生命周期、明确输入输出和状态机的工作可以演进为普通 Worker；只有需要独立目标、决策与权限的自治 Agent，才适用第 13 章的准入条件。

| 能力 | 状态 | 边界说明 |
|---|---|---|
| WebSocket 对话入口 | 已实现、条件启用 | 依赖 LLM；逻辑会话尚无独立 turn ID、顺序和跨连接/实例并发所有权，不能安全承诺并发多轮 |

| 混合检索 | 已实现基础链路 | 帖子语义/关键词与异构实体候选仍需完善去重、配额和评测 |
| 会话记忆 | 已实现 | Redis 是跨进程数据，Caffeine 只加速热会话 |
| Trace/Observation | 已实现基础链路 | 同一连接复用 correlation ID，而 Trace 表对该字段有唯一约束，多轮持久化可能失败且不阻塞回答；失败模式查询目前也未覆盖全部终态 |
| Agent 离线回归 | 已实现基础框架 | Agent 与 Judge 当前获得的场景信息不完全对称，修正前不能作为可信基线 |
| 内容质量评估 | 已实现基础服务 | 与 Agent 离线回归框架不是同一套机制 |
| 创作任务状态机 | 未实现 | 当前没有正式草稿任务、审批和版本恢复合同 |
| Agent 写工具 | 未实现 | 不允许把只读 Tool 合同直接当作安全写入能力 |
| 统一后台任务编排 | 未实现 | 现有后台能力尚未统一为持久任务平台 |
| 多 Agent 路由与协作 | 未实现、当前非目标 | 等待稳定自治边界和评测证据 |

## 11. 可靠性、安全与质量约束

### 11.1 执行边界

- 每次同步执行必须有最大步数、工具超时、模型超时和总 deadline。
- 取消是协作式信号，不能假设 `Future.cancel(true)` 一定终止底层调用。
- 未知工具、非法参数和解析失败不得扩大权限或进入无限重试。
- 阶段 0 采用“每个逻辑会话 `(userId, chatSessionId)` 同一时刻最多一个 in-flight turn”；后续请求必须明确拒绝或排队，并以跨连接、跨实例的原子所有权保证串行，不能只在单个 WebSocket 对象内加锁。
- 长时任务必须脱离连接生命周期，使用持久状态与重放机制。

### 11.2 权限与副作用

- 身份和资源权限由服务端确定，不接受模型文本声明。
- 外部内容、站内文章和工具输出都是不可信数据，必须与系统指令分隔。
- 最终回答阶段必须重新注入不可变系统约束；页面、检索和工具 evidence 以结构化来源、长度预算和隔离边界进入，不能退化为无来源的普通用户文本。
- 高风险写入需要预览、确认、幂等和审计。
- 主动行为应有频率限制、用户控制和关闭入口。

### 11.3 隐私与记忆

- Memory、Trace 和长期状态分别定义保留期、访问权限与删除策略。
- Trace 使用字段白名单和摘要，避免无边界持久化原始上下文与工具结果。
- 长期记忆的写入应可解释、可纠错、可由用户删除。

### 11.4 质量准入

能力进入主链前至少需要：

- 代表性单元测试与 Core-only 装配测试；
- 工具契约和失败降级测试；
- 固定数据集上的离线回归；
- 延迟、错误率、token/成本与依赖超时观测；
- 对提示注入、无答案和权限绕过的对抗测试。

## 12. 当前优先改进项

在增加创作、多 Agent 或更多人格扩展前，优先关闭当前主链的正确性缺口：

1. **P0——修复最终回答的约束与证据边界**：重新注入不可变系统约束；将页面、检索和工具 observation 标记为不可信 evidence，并定义来源、长度预算与隔离格式，避免完整 Context 只进入首次决策。
2. **P0——建立可靠的 turn 身份与串行语义**：固定 ID 合同——`connectionId` 标识传输连接，服务端生成的 `turnId` 同时作为该轮根 correlation ID，客户端 `requestId` 用于请求去重，子步骤以 `parentId/causationId` 关联；修复当前连接级 correlation ID 被 Trace 唯一键拒绝的问题。阶段 0 以 `(userId, chatSessionId)` 为隔离键，通过跨连接、跨实例的原子所有权保证单 in-flight，后续请求拒绝或排队，并定义 sequence 与断连取消行为，暂不实现多路复用。
3. 对帖子语义结果先做 chunk-to-document 聚合，再进行 RRF；区分帖子与 Bangumi 实体的评测口径。
4. 将检索单路异常、超时、真实空结果和代码错误分开观测，并增加独立 deadline。
5. 修正失败模式挖掘的状态查询范围，让 TIMEOUT、ABORTED 与 FAILURE 规则真正可达。
6. 让 Agent 与 Judge 使用同一份只读场景快照，修正会话历史、页面上下文和工具证据不对称，再建立可复现基准。

这些问题直接影响读者协作的事实性、归因与可靠性，其优先级高于引入更多 Agent 角色。

## 13. 多 Agent 演进触发条件

多 Agent 不是功能数量达到阈值后的自然结果。拆分前必须满足全部硬前提：

1. **目标与输入输出可独立**：子任务有独立完成定义，不依赖整段隐式上下文。
2. **状态所有权稳定**：每个角色拥有明确任务状态，不会无版本控制地并发修改同一草稿或记忆。
3. **失败可恢复或补偿**：局部失败可以重试、跳过或回滚，不让整个工作流进入未知状态。
4. **Trace 可贯通**：父任务和子任务具有 correlation/causation 关系，成本、延迟和失败可以归因。
5. **单 Agent 基线可信且稳定**：已有基准证明瓶颈来自任务拓扑，而不是检索、Prompt、工具或实现缺陷。

满足硬前提后，还需至少一个收益触发器达到实验前声明的量化门槛：

- **并行收益**：并行检索、审校或分析对成功率或端到端延迟的改善，大于通信 token、调度成本和新增故障面。
- **权限隔离**：角色确实需要不同工具、数据范围或审批流程，拆分能够实质缩小权限而不是只改变命名。
- **独立评价与发布价值**：子任务能独立衡量、灰度和回滚，并带来可复现的质量收益。

门槛不在本文固定为一个通用百分比。每次实验必须先写明基线、目标成功率/延迟/成本、允许的回归项与样本规模，再看结果决定是否固化拓扑，不能在看到结果后改判定标准。

可能的未来拆分不是“读者 Agent、创作 Agent、后台 Agent”简单一一对应，而是沿稳定责任拆分，例如：

- Retrieval Worker：只负责多源召回和证据标准化；
- Draft Worker：只基于批准证据生成草稿；
- Verification Worker：独立检查事实、引用与约束；
- Orchestrator：拥有任务状态、预算、审批和终止权。

这类角色只有在任务状态机和评测指标存在后才有意义。

## 14. 分阶段路线图

### 阶段 0：巩固读者协作主链

- 关闭第 12 章列出的上下文、检索、WebSocket 和评测缺口；
- 建立可复现的检索与回答基准；
- 为引用、无答案拒答和提示注入增加测试；
- 完成同步 turn 的总 deadline 与取消语义。

完成标准：每轮 turn 可唯一追踪，且同一 `(userId, chatSessionId)` 在跨连接、跨实例情况下也不会并发串扰；Agent 与 Judge 使用同一场景快照；读者任务在固定数据集上可重复评估，失败能定位到 Context、Retrieval、Tool 或 LLM。

### 阶段 1：受约束创作协作

- 引入创作任务、约束、证据和草稿版本模型；
- 增加大纲/审校等无副作用能力；
- 增加仅写入草稿区的受控写工具，并设计预览、审批、幂等和回滚；
- 将任务状态与普通聊天历史分离。

完成标准：用户可以看到任务计划、证据、草稿差异和审批点；Agent 可以安全写入可恢复的草稿版本，但不能直接发布或修改正式内容。

### 阶段 2：持久后台智能任务

- 抽象持久任务状态机、Worker、重试和死信；
- 首先让内部离线评测与索引复用统一任务和 Trace 合同；
- 其次接入知识抽取与内容治理，并增加配额、调度、人工审计与结果回放；
- 最后再评估面向用户的长时资料整理，避免过早扩大产品承诺。

完成标准：浏览器断开或服务重启后任务可恢复，结果有输入快照与版本证据。

### 阶段 3：按证据评估多 Agent

- 选择具备稳定输入输出的子任务做单点实验；
- 与单 Agent/单 Worker 基线比较成功率、延迟、成本和恢复性；
- 先满足第 13 章全部硬前提，再验证至少一个收益触发器；只有达到预设门槛且不扩大安全边界时，才固化多 Agent 拓扑。

完成标准：拆分带来的收益由基准证明，子任务权限、状态、失败补偿和 Trace 均可独立验证。

## 15. 成功指标

### 15.1 读者协作

- 检索 Recall@K、MRR/nDCG 与无结果率；
- 引用正确率、groundedness、无答案拒答率；
- 工具选择准确率、平均无效步骤数；
- 首 token、总延迟、超时率和单轮成本；
- 用户追问完成率和纠正率。

### 15.2 创作协作

- 任务完成率与人工接受率；
- 约束满足率、事实错误率和引用覆盖率；
- 用户修改量、回滚率和平均审批轮次；
- 每个任务的步骤、时间与成本。

### 15.3 后台智能任务

- 任务吞吐、排队时间、成功/重试/死信比例；
- 服务重启后的恢复率；
- 单任务资源和模型成本；
- 人工复核通过率与错误影响范围。

### 15.4 多 Agent 实验

- 相对单 Agent 基线的成功率增益；
- 端到端延迟和 token/成本变化；
- 失败归因完整率与补偿成功率；
- 权限扩大、状态冲突和循环失控次数。

没有基线与目标阈值时，不以“角色数量”或“调用次数”作为进展指标。

## 16. 已确认决策

| 决策 | 结论 |
|---|---|
| Agent 产品角色 | 内容社区内的情境协作者，不是外挂聊天框 |
| 当前核心能力 | 读者协作 |
| 下一阶段 | 受约束创作协作 |
| 后台能力 | 作为持久智能任务演进，不绑定 WebSocket 生命周期 |
| 研究能力 | 保留为扩展，暂不作为核心产品层 |
| 当前拓扑 | 单 Agent Core + Contributor + Tool + 扩展组 |
| 多 Agent | 第 13 章全部硬前提满足后，至少一个预声明的量化收益触发器达标才评估固化 |
| Code Agent 关系 | 借鉴受约束执行机制，不复制产品定位和权限模型 |
| AI 可用性 | 条件能力，关闭后主站仍可运行 |
| 质量原则 | 先修正确性、可恢复性和评测，再增加角色与功能 |

## 17. 非目标

本文不决定：

- 具体模型供应商或长期固定模型；
- Prompt 的逐字内容；
- UI 视觉样式与 Live2D 资源；
- 创作任务和后台任务的最终数据库表结构；
- 多 Agent 框架选型；
- 尚无基准数据支撑的性能与质量目标值。

这些内容应在对应阶段形成独立设计，并继续遵守本文的产品定位、权限边界与演进准入条件。
