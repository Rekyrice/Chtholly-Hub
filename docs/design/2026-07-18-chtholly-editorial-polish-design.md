# Chtholly Hub 编辑型页面与珂朵莉房间精修设计

> 日期：2026-07-18
> 状态：待文档复核
> 适用范围：Hub 热门推荐、文章详情、搜索页、Chtholly 页面及主题聚合状态

## 1. 背景与问题

现有页面已经建立统一的大图背景、刘海标题区、玻璃面板和珂朵莉角色入口，整体识别度不应推倒重来。本轮需要解决的是信息结构和组件密度不够精确的问题：

- Hub 热门推荐的标题与摘要可用空间不足，不同文章切换时内容被截断，卡片高度与控制区互相争抢。
- 文章详情主栏偏窄，标题区、信息区和正文像多个白框拼接；侧栏内容少，无法形成稳定的阅读辅助。
- Chtholly 页面仍像数个业务卡片顺序排列，“最近在想什么”没有使用已有的重要度、来源等数据表达层级，“她注意到的主题”无法区分“暂无聚类”与“接口失败”。
- 搜索页复用了首页文章卡片，单条结果过高；已有补全、标签筛选和游标分页能力没有在界面暴露。
- 主题聚类任务虽然成功执行，但空数组同时承担“确实没有主题”“数据尚未生成”“任务失败”三种语义，前端无法给出准确反馈。

本轮视觉方向确定为“安静书桌”：保留通透背景与角色气质，用更稳定的版心、较少但更明确的层级、纸张与便笺式细节增强编辑感，不引入夸张悬浮岛或大规模导航重构。

## 2. 目标

- 让热门推荐在桌面端完整承载三行标题与三行摘要，任何轮播项切换都不改变外框高度。
- 将文章详情主栏扩大到适合长文阅读的比例，并建立由阅读进度、目录、文章线索和珂朵莉陪伴组成的分块侧栏。
- 把 Chtholly 页面从“同权卡片列表”改成具有主次关系的房间叙事：今夜手记、轻量时间线、窗边便笺和书架。
- 让主题区域准确表达 `READY`、`SPARSE`、`PENDING`、`FAILED` 四种状态，不伪造主题内容。
- 将搜索页改成高密度的编辑型检索工作台，接入已有补全、标签和游标分页能力。
- 保留现有刘海结构、全页背景、主导航、聊天入口和角色图片，不改变文章业务及 Agent 会话主流程。

## 3. 非目标

- 不重新设计站点导航、登录流程、写作器、Admin 或 Agent 工作台。
- 不替换本轮范围之外的页面背景图，也不修改用户已指定的图片分配。
- 不将文章正文改成全宽排版，不牺牲长文行长与可读性换取表面上的“更宽”。
- 不把标签热度、推荐文章或静态文案伪装成由聚类算法产生的主题。
- 不为搜索引入新的搜索引擎或重写 Elasticsearch 查询链路。
- 不删除现有 Chtholly 内嵌聊天；房间内容与聊天入口继续共存。

## 4. 设计原则

### 4.1 同一背景，不同材质

全页继续使用当前路由背景和蒙版。内容层只使用一套冷白玻璃材质变量，差异通过边框、留白、分隔线和浅蓝纸张层表达，不为每个模块单独换色。

### 4.2 高度由结构决定，不由内容碰运气

轮播卡片、搜索结果和 Chtholly 分区都先定义稳定的行轨道与最小高度，再限制文本行数。导航控件不得绝对定位覆盖正文。

### 4.3 编辑感来自秩序

参考编辑型博客的日期、作者、栏目、摘要和正文节奏，但不复制其视觉皮肤。标题是第一层，元数据是第二层，说明与角色提示是第三层；辅助文字必须明显小于正文。

### 4.4 空状态也是真实状态

没有聚类结果时明确说明近期信号不足；请求失败时提供重试；任务尚未生成时显示等待。空状态不使用虚构主题填满版面。

## 5. Hub 热门推荐

### 5.1 桌面结构

`HubPersonalizedRecommendations` 保留现有标题和轮播语义，内部改为四段式结构：

```text
区域标题
┌──────────────────────── 推荐主体 ────────────────────────┐
│ 58% 封面                         42% 文章信息             │
│                                  栏目 / 作者              │
│                                  三行标题                 │
│                                  三行摘要                 │
│                                  文章日期                 │
└──────────────────────────────────────────────────────────┘
轮播进度点                              上一篇  1/5  下一篇
```

- 推荐主体桌面端固定为 `390px` 高，封面与内容列同高；整个推荐模块约 `500px`，允许比当前版本更高。
- 桌面端列比例为 `58:42`，内容列水平内边距 `32px`、垂直内边距 `30px`。
- 标题保留三行轨道，字号使用 `clamp(1.8rem, 2.6vw, 2.55rem)`，行高约 `1.12`；只在第四行发生省略。
- 摘要直接展示文章摘要，不添加“珂朵莉说”等角色前缀；保留三行轨道，字号与正文辅助层一致。
- 日期放在摘要下方独立一行，不与轮播控制共用空间。
- 进度点与前后按钮位于推荐主体下方的独立控制行，不覆盖标题、摘要或图片。
- 所有轮播项、加载骨架和失败占位使用相同主体高度，切换时外框不得产生纵向位移。

### 5.2 响应式行为

- `768px–1099px`：比例调整为 `52:48`，主体最小高度 `370px`，标题仍预留三行。
- `<768px`：改为上下结构，图片使用固定纵横比，文本区不使用固定整体高度；控制行仍独立存在。
- `<480px`：标题最多三行、摘要最多两行，以避免首屏被单张推荐完全占满。
- `prefers-reduced-motion` 下停用自动淡入位移动效，只保留即时切换或低幅透明度变化。

### 5.3 验收标准

- 使用最长现有标题和摘要时，桌面端标题至少完整显示三行，摘要至少完整显示三行。
- 从任意推荐切到下一条，模块外框高度变化不超过 `1px`。
- 控制区与正文在所有断点下无重叠。
- 推荐摘要中不出现前端追加的角色口吻前缀。

## 6. 文章详情

### 6.1 版心与主卡片

桌面端总版心保持 `1260px`，调整为：

| 区域 | 宽度 |
|---|---:|
| 文章主栏 | `920px` |
| 栏间距 | `28px` |
| 阅读侧栏 | `312px` |

文章主栏不再由标题白框、信息白框和正文白底拼接，而是使用一张连续的编辑纸面：封面之后依次为标题、日期/作者/栏目、操作区、导语、正文和文末信息。可以用细分隔线或轻微透明度变化划分段落，但不得出现突兀的纯白横断面。

- 主卡片顶部在封面结束后保留 `48–56px` 呼吸区，标题不紧贴上边界。
- 正文内容列最大宽度 `760px`，在 `920px` 主栏内居中，保证约 `65–80` 个中文字符的视觉行长。
- 正文图片允许扩展至内容列宽；图片说明使用 `0.78rem–0.84rem`、居中、较低对比度，与正文至少留 `12px` 间距。
- 文章日期同时出现在标题元数据和外部文章浏览卡片中，统一使用本地化绝对日期，不以悬停信息代替。

### 6.2 阅读侧栏

`ArticleReadingSidebar` 保留组件边界，内部拆为四个可独立降级的分块：

1. **阅读轨迹**：阅读进度、发布日期、预计阅读分钟数、正文字符数。
2. **本页目录**：使用现有 Markdown heading 提取结果；滚动时高亮当前二级/三级标题。短文无有效标题时展示两到三条文章摘要线索，不渲染空目录。
3. **文章线索**：作者、标签以及最多两篇 `postService.related(id)` 返回的紧凑相关文章。相关推荐失败不影响正文。
4. **珂朵莉陪伴**：保留现有角色插图和“问珂朵莉”入口，作为侧栏末段，不抢占目录首屏位置。

侧栏桌面端 `position: sticky`，顶部偏移与站点导航一致；每个分块使用同一种玻璃材质，通过 `16–20px` 间距分开。平板端侧栏移到正文后方，移动端不使用 sticky。

### 6.3 数据与失败边界

- 页面服务端并发获取文章详情和 `postService.related(id)`；相关文章失败返回空数组，不将详情页整体升级为错误页。
- 预计阅读时间和字符数由服务端基于清理后的 Markdown 文本计算，避免首屏 hydration 后数值跳变。
- 当前标题高亮属于客户端小岛，只监听文章容器内的 heading，不改变页面级滚动位置。

### 6.4 验收标准

- `1440px` 及以上视口中，主栏为 `920px`、侧栏为 `312px`，无横向溢出。
- 标题区到正文为连续表面，正文首段与容器上边界至少相距 `40px`。
- 图片说明明显小于正文且水平居中。
- 任意侧栏接口失败时文章正文仍可阅读。

## 7. Chtholly 房间

### 7.1 空间叙事

保留当前 `ChthollyRoomHero`、背景 `1031747.jpg`、角色占位和内嵌聊天入口。Hero 之后不再使用四个同权矩形，而使用“书桌—窗边—书架”的纵向叙事：

```text
┌──────────────── 今夜书桌（主区域） ────────────────┐
│ 今夜手记（主内容）            轻量经历时间线          │
└───────────────────────────────────────────────────┘
┌──────────── 窗边便笺（主题） ──────────────────────┐
└───────────────────────────────────────────────────┘
┌──────────── 书架（推荐） ──────────────────────────┐
└───────────────────────────────────────────────────┘
```

不通过木纹贴图、拟真家具或大量装饰物制造“房间”，而通过纸张层、细线、页角、日期戳和留白表达安静书桌。全页仍保持现有清晰纵深和背景可见度。

角色气质通过克制的细节体现：冬青蓝作为交互色，红色只作为极少量印章或宝石提示，正文仍使用中性深色；文案保持平静、直接、略带温柔，不堆叠卖萌语气、角色口癖或装饰图标。这样既能辨认珂朵莉，又不会把页面做成主题乐园式角色陈列。

### 7.2 “她最近在想什么”改为“今夜手记”

`ChthollyExperienceTimeline` 使用现有 `importance`、`valueScore`、`source` 和时间字段重排：

- 首条选择规则：按 `importance` 降序，其次按 `valueScore`、时间降序；它成为一张较大的“今夜手记”。
- `source=community-quiet` 不作为与思考等高的正文卡片，而显示为手记旁的一条低对比状态注：“今晚社区很安静”。
- 其余有效经历按时间倒序形成细线时间轴，每条只展示时间、短标题和一至两行摘要；默认显示三条，可展开其余内容。
- 周总结若存在，表现为一封独立的“本周来信”，放在时间轴末尾；没有周总结时不保留空框。
- 不改变原始文本，不将重要度数值直接展示给普通用户；数值只用于布局排序和视觉权重。

### 7.3 “她注意到的主题”改为“窗边便笺”

`ChthollyTopicWindow` 根据主题概览状态渲染：

- `READY`：每个主题是一张小便笺，展示主题名、文章数、更新时间和最多三个关键词。
- `SPARSE`：显示“近几天还没有形成稳定的话题”，并提供真实的热门标签信号，但清楚标注为“近期标签”，不得称为聚类主题。
- `PENDING`：显示正在整理近期内容及上次更新时间，不渲染空白大框。
- `FAILED`：显示轻量错误说明和重试按钮；不得把失败伪装成“社区安静”。

主题区高度由内容决定，不再被右列 `1fr` 强制拉到与经历区相同高度。桌面布局改为主题区 `auto`、推荐区填充剩余空间，左右底边仍通过整体网格对齐。

### 7.4 推荐改为“书架”

`ChthollyRecommendationShelf` 使用横向或双列紧凑书脊式条目：封面缩略图、标题、作者、日期和一句摘要。推荐是可浏览内容，不再表现为第三组大帖子卡片。桌面端每行两条，平板与移动端单列。

### 7.5 主题数据契约

保留现有 `GET /api/v1/topics` 列表接口兼容性，新增非破坏性的 `GET /api/v1/topics/overview`：

```json
{
  "items": [],
  "state": "SPARSE",
  "lastAttemptAt": "2026-07-18T00:00:15+08:00",
  "lastSuccessAt": "2026-07-18T00:00:15+08:00",
  "windowDays": 7,
  "reason": "INSUFFICIENT_SIGNALS"
}
```

状态定义：

| 状态 | 条件 |
|---|---|
| `READY` | 最近一次成功聚类且 `items` 非空 |
| `SPARSE` | 最近一次任务成功，但符合聚类阈值的内容不足 |
| `PENDING` | 尚无任务记录，或缓存缺失且正在异步重建 |
| `FAILED` | 最近一次尝试失败，且没有可用的旧结果 |

- 应用启动后只在缓存缺失或状态过期时异步触发一次重建，并继续复用现有分布式锁，不能阻塞 Spring 启动。
- 保持“至少两篇内容形成主题”的真实性约束；标签兜底只允许完全相同的规范化标签将至少两篇文章归组，不因页面需要而降低为单篇主题。
- 聚类为空时可返回独立的热门标签信号供 `SPARSE` 状态展示，但该字段与 `items` 分离。
- 任务运行记录必须区分尝试时间、成功时间与结果状态，不能只记录一个 `success=true`。

## 8. 搜索工作台

### 8.1 信息结构

搜索页保留当前背景、刘海和侧栏位置，主栏改为：

1. 带自动补全的搜索输入。
2. 可移除的筛选条：当前关键词、标签、排序方式。
3. 结果统计与排序行。
4. 专用紧凑结果列表。
5. 使用游标的“加载更多”。

初始未搜索状态不再只有一个空白输入框，而展示“最近搜索入口、热门标签、近期更新文章”三种真实入口；没有本地搜索历史时只显示后两项。

### 8.2 专用搜索结果卡

新增搜索结果组件，不复用首页 `PostCard`：

- 桌面端单条高度约 `188–220px`，左侧 `240px` 封面，右侧依次为栏目/日期、两行标题、两行摘要、作者和标签。
- 无封面时使用现有统一占位，不放大成整张色块。
- 搜索关键词高亮只作用于后端返回或前端安全切分后的纯文本，不通过不可信 HTML 注入。
- 日期必须在卡片主信息区可见。
- 移动端改为上图下文，仍显式保留日期和作者。

### 8.3 已有能力接入

- 输入稳定 `250ms` 后调用现有 `searchService.suggest(prefix)`；键盘上下键可选择，`Escape` 关闭，输入法组合期间不发送请求。
- 搜索请求透传标签参数；筛选改变后重置游标和现有结果。
- 首屏由服务端请求，后续通过 `nextAfter` 与 `hasMore` 追加，避免将全部结果一次渲染成长页面。
- 加载更多失败时保留现有结果，并在按钮原位提供重试。

### 8.4 空状态

- **无关键词**：热门标签与近期文章。
- **有关键词但无结果**：保留原查询，给出删除某个筛选、检查拼写和查看相近补全词三类行动。
- **服务降级**：明确搜索暂不可用，不将其表述为“没有文章”。

### 8.5 验收标准

- 桌面端 18 条结果的主列表高度不超过约 `4500px`，不再使用首页大卡片造成超长页面。
- 补全支持鼠标与完整键盘操作，焦点移动可被屏幕阅读器识别。
- 标签筛选与加载更多均真实改变请求参数，并能从 URL 恢复当前搜索状态。
- 任一追加请求失败不会清空已经显示的结果。

## 9. 组件与数据边界

前端建议保持 Server Component 负责首屏数据，Client Component 只承载交互：

| 页面 | 服务端职责 | 客户端小岛 |
|---|---|---|
| Hub | 获取推荐首屏 | 轮播切换与自动播放 |
| 文章详情 | 详情、相关文章、阅读统计初值 | 阅读进度、目录高亮、文章操作 |
| Chtholly | 经历、主题概览、推荐并发加载 | 展开时间线、主题重试、内嵌聊天 |
| 搜索 | 根据 URL 获取首屏和筛选基础数据 | 自动补全、筛选交互、游标追加 |

新增类型必须落在 `apps/web/lib/types`，接口调用必须落在 `apps/web/lib/services`；页面组件不得直接拼接 API URL。后端主题状态使用独立 DTO，不把 Redis 内部键或调度实现暴露给前端。

## 10. 可访问性与动效

- 所有轮播、补全、折叠时间线和加载更多控件具备可见焦点与明确 `aria-label`。
- 轮播自动播放在鼠标悬停、键盘焦点进入或页面不可见时暂停。
- 文本与玻璃背景的对比度以正文至少 `4.5:1` 为目标；不得仅靠阴影挽救浅色文字。
- `prefers-reduced-motion` 下关闭自动滚动、位移和粒子增强，仅保留必要状态变化。
- 客户端加载与主题重试不得抢夺页面滚动位置或输入焦点。

## 11. 测试与验证

### 11.1 前端自动化

- 热门推荐：最长标题/摘要、固定高度、无角色前缀、移动端结构和控制区顺序。
- 文章详情：主/侧栏结构、短文目录回退、相关文章失败降级、图片说明类名。
- Chtholly：重要度排序、`community-quiet` 状态化、四种主题状态、推荐书架结构。
- 搜索：补全防抖、输入法组合、标签透传、游标追加、失败保留已有结果、URL 状态恢复。
- 运行 `npm run test:run` 与 `npm run build`。

### 11.2 后端自动化

- 主题概览四种状态的映射。
- 成功空结果与执行失败的区别。
- 缓存缺失时的异步启动刷新、分布式锁复用和非阻塞行为。
- 相同规范化标签至少两篇才形成兜底主题，单篇不得成为主题。
- 运行主题领域定向测试，最终运行 `mvn test`。

### 11.3 视觉核验

在 `1440×900`、`1024×768`、`390×844` 三个视口验证：

- 推荐轮播各条目切换无纵向跳动。
- 文章标题区、正文和侧栏对齐，图片说明层级正确。
- Chtholly 的今夜书桌、窗边便笺和书架具有主次关系，但仍使用同一站点材质。
- 搜索结果密度显著高于首页文章卡片，空状态与错误状态可区分。
