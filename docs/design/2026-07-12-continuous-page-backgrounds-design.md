# Chtholly Hub 全页连续背景设计

## 1. 目标

将普通站点页面从“刘海一张图、正文另一张图”改为“每个页面只有一个真实全页背景图层”。刘海继续保留现有高度、标题、打字机文字、粒子、局部蒙版和底部羽化，但不再渲染独立图片。正文组件继续覆盖在同一个背景之上，从而消除刘海与正文之间的图片接缝和构图跳变。

本次只调整页面背景体系和必要的蒙版、透明表面适配，不修改推荐框、侧栏、文章卡片、写作表单、字体和业务交互结构。

## 2. 已确认的视觉原则

1. 每个普通页面只有一个固定在视口后方的图片背景；滚动正文时不切换为第二张图。
2. 刘海只是覆盖在全页背景上的结构层，不持有图片资源。
3. 刘海区域通过较深的局部蒙版保证标题和打字机文字清晰；正文通过较浅的全页蒙版及半透明组件保证阅读性。
4. Hub 是唯一例外：三张全页背景与三条 Hero 打字机文案按相同索引顺序轮换，并在文案进入下一条时交叉淡入。
5. `prefers-reduced-motion: reduce` 下保留索引同步，但取消背景交叉动画，直接切换。
6. Agent 工作台保持现有壁纸轮换机制和现有图片不变。
7. 珂朵莉房间页面保持现状，等待后续独立重设计。

## 4. 背景架构

### 4.1 单一背景所有权

`SiteChrome` 继续负责站点外壳，并成为普通页面背景的唯一所有者。背景组件位于导航栏下方、刘海和正文之前，使用 `position: fixed` 覆盖可视区域。刘海与正文都透明地显示这一个背景，不再分别设置 `background-image`。

路由视觉配置从“共享刘海配置 + 页面配置”收敛为页面级配置：

- `images`：一个或多个正式背景资源；除 Hub 外长度均为 1。
- `positionDesktop`、`positionMobile`：响应式焦点位置。
- `overlayAlpha`、`blurPx`、`saturate`：全页基础处理。
- 刘海局部蒙版仍由 `SiteHeader` 的 CSS 负责，但不包含图片地址。

Admin 纳入页面视觉配置。无配置的未知页面继续使用无图渐变降级；404 通过显式视觉覆盖把指定图片传给 `SiteChrome`。该背景从导航栏下方固定覆盖整个剩余视口，刘海结构和 404 内容共同显示它，404 内容组件自身不再创建图片背景，从而避免“渐变刘海 + 另一张 404 背景”的双背景。

### 4.2 Hub 与打字机同步

当前 `useTypewriterSequence` 已维护文案索引，但只返回文本。实施时让它同时返回当前索引，并由 `HeroTypewriter` 将索引变化通知 `SiteHeader`，再由 `SiteChrome` 更新 Hub 全页背景索引。

同步规则如下：

1. 初始文案索引 0 对应 `1031784.jpg`。
2. 当前文案完成停留并擦除后，文案索引进入下一项；此时背景同步进入下一项。
3. 三条文案与三张图片均按 `0 → 1 → 2 → 0` 循环，不再维护第二套计时器。
4. 背景组件保留前后两个图层完成交叉淡入；过渡结束后只有当前图层可见。它们属于同一个全页背景组件，不会在刘海边界重新裁切。

### 4.3 特殊页面

- Landing 保持现有独立全屏结构，只替换为指定正式资源；不接入 `SiteChrome`。
- Write 保持无刘海的专注写作布局，但其正文后方使用指定全页背景。
- Agent 完全不改动现有 `AgentPageBackground` 和 `useWallpaperRotation`。
- Chtholly 完全不改动当前房间分区。
- 404 删除自身的第二套图片背景，改由外层 `SiteChrome` 提供唯一固定全页背景；404 文案和插图只作为透明内容层覆盖其上。

## 5. 素材处理

1. 不从 `_incoming` 直接引用运行时 URL，也不使用 `git add -f`。
2. 正式资源使用语义化文件名放入 `apps/web/public/images/site/backgrounds/`；Landing 可放入 `apps/web/public/images/landing/`。
3. 转换时保留原图宽高比和构图，不做内容裁剪；页面焦点只通过 CSS `background-position` 调整。
4. 输出 WebP 并做网页压缩。超高分辨率图片允许等比限制最长边以控制解码和传输成本，但不改变画面比例。
5. Hub 三张源图宽高比分别约为 1.4146、1.4147、1.4147，虽像素尺寸不完全相同，但 CSS `cover` 下不会产生布局跳动；不为“统一尺寸”强制裁剪。
6. 替换完成后删除已无引用的旧正式背景，保留仍被 Chtholly 使用的 `landing/default.jpg`，避免间接改变留空页面。

## 6. 蒙版与可读性

1. 全页背景蒙版维持较低强度，让图片保持可辨识。
2. 刘海在全页背景之上增加独立暗色渐变，标题和打字机文字必须通过桌面、移动截图验证。
3. 正文卡片继续使用现有透明表面 Token，不为不同页面设置不同色框。
4. 写作区域保留比普通卡片更高的不透明度，但背景仍应在编辑器外侧和大面积空白处明显可见。
5. 无 `backdrop-filter` 的浏览器继续使用现有高不透明度降级，优先保证文字可读。

## 7. 响应式与视觉验收

每张背景都要分别确定桌面和移动焦点位置。用户未指定必须保留的主体区域，因此以“人物头部、脸部和主要叙事物体不被刘海边界或正文首屏严重裁掉”为默认准则。

至少验证：

- 桌面：2048 × 1150。
- 移动：390 × 844。
- Hub：三张背景与三条文案索引一致，刘海到正文没有图片跳变。
- Search、Login、About、User、Settings、Archive、Tag、Post、Admin：刘海和正文始终显示同一张图。
- Write：无刘海，背景可见且编辑器文字清楚。
- Landing：背景与分享元数据指向同一正式资源。
- 404：指定图片从导航栏下方覆盖整个页面，不再出现额外渐变刘海、局部白底或第二张背景。
- Agent、Chtholly：视觉和行为与本次修改前一致。

## 8. 测试与交付

1. 先修改测试锁定单背景所有权、路由映射、Hub 索引同步、404 覆盖和特殊页面排除规则。
2. 运行相关 Vitest，再运行完整 `npm run test:run` 和 `npm run build`。
3. 使用真实浏览器完成桌面、移动和 Hub 轮换截图验收。
4. 提交前及 push/PR 前分别执行仓库规定的 Git ignore 新增文件审计。
5. PR 合并前保留任务 worktree；用户确认合并后再按仓库流程清理。

- 不修改字体、推荐框内容、侧栏信息架构、写作业务流程或后端 API。
