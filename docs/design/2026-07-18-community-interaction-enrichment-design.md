# 社区互动真实性增强设计

## 1. 背景与现状

公开资料一致性和第一版内容包已经完成，当前站点共有 44 篇公开文章，其中 40 篇由 `content-v3` 管理。八个社区账号已经具备头像、简介、标签和文章，但社区行为仍明显像一次性初始化：

- 内容包只有 36 条有效评论（28 条顶级评论、8 条回复），约 18 篇公开文章完全没有可见评论。
- 评论大多集中在 7 月 14 日至 15 日，与文章从 3 月到 7 月的发布时间跨度不匹配。
- 内容包只有 62 个点赞、26 个收藏和 14 条关注关系，账号之间关系较薄，站长账号也没有来自社区账号的自然关注。
- Redis 中目标文章现有 91 个点赞位和 91 个收藏位；其中分别有 27 个、64 个位属于 MySQL 已不存在的用户，导致展示计数和有效成员关系不一致。
- 现有导入器通过普通点赞/收藏服务产生异步事件和通知，随后又立即从尚未收敛的 SDS 重建用户计数，可能把正确计数覆盖成旧值。
- MySQL 已有正式备份脚本，但点赞/收藏成员关系只存在于 Redis，仓库没有对应备份入口。
- 审阅页能逐条展示互动，但没有自动测试，也不能一眼看出零评论文章、账号参与分布和时间分布。

这些问题不能靠一次性 SQL 或简单把数字抬高解决。目标是让互动内容、时间、人物关系和运行时计数同时可信，并让整套数据可以重复导入、核对和恢复。

## 2. 目标

1. 让全部 44 篇公开文章至少形成一段可读的讨论，同时保留冷热差异。
2. 让八个账号的评论、点赞、收藏和关注符合各自兴趣，并呈现不均匀的长期行为轨迹。
3. 允许社区账号关注站长 `Rekyrice`，但绝不代表站长账号执行任何行为。
4. 将内容包拥有的点赞/收藏视为精确期望状态，幂等重跑不会重复累计或制造通知。
5. 清理目标文章上属于不存在用户的孤儿点赞/收藏位，并从 bitmap 精确重建展示计数。
6. 在用户计数、缓存和搜索索引更新前完成互动事实对账，消除异步聚合竞态。
7. 为内容包互动数量、覆盖率、层级、时间和分布建立可执行合同。
8. 在正式导入前同时备份 MySQL 与 Redis，并保留 dry-run、正式导入和幂等复跑证据。

## 3. 非目标

- 不新增账号、文章、头像或文章图片。
- 不重写已经验收的文章正文。
- 不替 `Rekyrice` 点赞、收藏、评论或关注任何账号。
- 不批量生成珂朵莉评论，也不把角色账号伪装成普通用户。
- 不增加持续自动运营、定时造互动或机器人刷量任务。
- 不给历史互动生成通知；导入不应打扰真实用户。
- 不在本次补做用户“评论历史”“点赞历史”等尚未接入的前端功能。
- 不操作其他分支或 worktree。

## 4. 方案比较

### 方案 A：完整内容包加精确运行时对账（采用）

继续以 `content/seed/content-v3/interactions.yml` 作为社区互动的 Git 权威清单。MySQL 中的评论和关注仍由稳定 identity 管理；Redis 中由内容包账号产生的点赞/收藏按清单精确设置；目标文章上的无效用户位被清理；SDS、用户计数、缓存和搜索索引从权威事实重建。

优点是可审阅、可重复、可验证，既保留真实用户状态，又能修复当前计数漂移。代价是需要补一个严格限定用途的历史互动维护服务。

### 方案 B：一次性 SQL 和 Redis 命令

直接向评论、关注表插入数据并改 Redis 数字，短期最快，但无法稳定重跑，也无法证明 bitmap、SDS、用户计数和搜索索引一致。后续环境重建时还会丢失操作记录，因此不采用。

### 方案 C：持续运行的自动互动账号

定时让子账号点赞、评论或关注，会产生更“活跃”的表象，但需要长期调度、内容审核和反滥用边界，且会混淆初始化数据与真实行为，不适合当前个人站点阶段，因此不采用。

## 5. 目标数据分布

### 5.1 评论与回复

目标为 96 条内容包评论：

- 72 条顶级评论、24 条一层回复，回复占 25%。
- 覆盖全部 44 篇公开文章，零评论文章为 0。
- 每篇 1 至 5 条，热点文章形成 2 至 3 个短线程，冷门文章只留一条具体观察。
- 每个账号 10 至 14 条，不能整齐平均；账号参与文章与其简介、标签和既有文章主题相符。
- 正文长度主要分布在 12 至 140 个中文字符，允许少量更短的口语回复和少量不超过 220 字的技术补充。
- 回复必须回应父评论里的具体细节；禁止“同意”“总结得很好”“很有启发”等无信息回复。
- 不复述标题或摘要，不使用“首先/其次/最后”“核心观点是”“值得我们思考”等模板化表达。

全部评论时间重新分散：最早不早于文章发布时间，最晚不晚于正式导入日；同一线程按顺序递增，同一账号不在短时间内批量出现。旧文章允许在发布后数日或数月被翻出，新文章以数小时至数日内互动为主。

### 5.2 点赞、收藏与浏览

- 点赞 168 个、收藏 80 个，`like:fav` 约为 2.1:1。
- 每个账号的点赞和收藏总量不同；收藏更集中于长文、技术实验和个人反复查阅的文章。
- 全部 44 篇文章都有至少一个点赞，但不要求都有收藏。
- 热点文章可有 6 至 8 个内容包点赞，冷门文章只有 1 至 2 个；禁止把同一组账号复制到所有文章。
- 44 条浏览最低值继续采用 minimum 语义，范围调整为约 48 至 420。旧文整体更高，但允许新番、热门技术故障文形成局部峰值。
- 浏览量不降低真实运行值，也不伪造逐次浏览事件。

### 5.3 关注关系

目标为 28 条受内容包管理的有效关注：

- 22 条社区账号之间的关系，包含少量互关，但不形成人人互关的完整图。
- 6 个社区账号关注站长 `Rekyrice`；其余两个账号不关注，保留差异。
- 每个社区账号有 2 至 4 个关注目标，入度和出度不平均。
- 禁止自关注、重复边和站长账号作为行为发起者。

## 6. 内容包合同

在 manifest 增加互动期望值：

- `expectedComments`
- `expectedRootComments`
- `expectedReplies`
- `expectedLikes`
- `expectedFavorites`
- `expectedFollows`
- `expectedViews`
- `expectedCommentedTargets`

运行时 Validator 和真实包合同测试同时检查：

1. 实际数量与 manifest 完全一致。
2. comment/reaction/follow seedKey 唯一。
3. 评论作者、文章目标和父评论都可解析。
4. 回复只有一层、父子同帖、创建时间晚于父评论和文章发布时间。
5. 每篇评论数不超过 5，每个账号参与量落在约定范围。
6. reaction 三元组 `(account, target, type)` 唯一。
7. follow 边唯一，来源只能是内容包账号，目标必须是内容包账号或配置中的站长。
8. 外部 slug 必须属于站长、公开且已发布。

`content-v3` 继续作为 schema/质量门禁版本，不为了本轮数据扩充改名为 `content-v4`。

## 7. 历史反应精确对账

新增只供维护/导入链路使用的反应事实对账服务，普通 API 继续走现有 `CounterService` 与事件通道。

正式导入对每个目标文章执行：

1. 解析八个受管理账号和 44 个目标文章的 ID。
2. 对八个账号的 `like`、`fav` 位逐一设置成 manifest 期望值；清单中不存在的受管理位被清除。
3. 扫描这些目标文章的 bitmap shard，解析所有置位用户 ID；批量查询 MySQL，清除只属于不存在用户的位。
4. 保留 `Rekyrice`、珂朵莉和任何仍存在的自然用户位，即使它们不在 manifest 中。
5. 在维护锁内按所有 bitmap shard 的 `BITCOUNT` 计算准确点赞/收藏总数。
6. 清理目标文章尚未刷新的 like/fav 聚合增量，原子写回 SDS 对应段；浏览段保持现值。
7. 对账完成后再重建作者的 `ucnt`，然后失效文章缓存并重建搜索文档。

该路径不发布点赞事件，所以不会生成历史通知，也不会与异步聚合出现“bitmap 已变、事件丢失”或“双计数”。失败时返回 `partial`，下一次以同一 manifest 重跑会继续收敛。

## 8. 外部关注目标

`SeedFollowDefinition` 支持以下二选一目标：

- `toAccountSeedKey`：内容包账号。
- `toHandle`：仅允许配置中的站长 handle。

`fromAccountSeedKey` 仍必须是内容包账号。Validator 在事务前验证目标二选一、禁止自关注和重复边；DatabaseWriter 将站长 handle 解析为唯一用户 ID，同时写 `following` 与 `follower` 镜像。稳定 follow identity 继续由 `(namespace, seedKey)` 管理。

导入结果要把外部目标用户 ID 纳入用户计数重建集合，使站长粉丝数与关系表一致。此设计只让社区账号关注站长，不给站长创建任何出站行为。

## 9. 评论与关注写入语义

- 评论和关注继续由 MySQL 事务及 `seed_content_identity` 管理。
- 完整 manifest 中删除的受管理评论软删除，删除的受管理关注将两侧关系设为无效。
- 写入历史评论和关注时不发布普通领域事件，因此不创建历史通知。
- 不删除、不覆盖同一文章上的真实用户评论和关注。
- 当前公开文章上两条内容为 `111`、`121` 的旧测试评论不属于内容包。正式导入前备份后，仅在文章、作者和正文三项精确匹配时软删除，并把匹配数与结果写入本地导入证据；不清理站长账号内容。

## 10. 备份、导入与回滚

### 10.1 备份

正式导入前必须生成同一时间窗口的两类仓库外备份：

1. 现有 `backup-mysql.ps1` 生成压缩 SQL、SHA-256 和 metadata。
2. 新增 `backup-redis.ps1` 通过 Redis 容器生成 RDB，复制到仓库外 D 盘，并生成 SHA-256 和 metadata。

两个脚本都拒绝相对路径、C 盘、仓库内部和符号链接逃逸。备份目标默认建议为 `D:\1.hhh\backups\Chtholly-Hub`。`.codex-tmp` 的导入快照和审阅页只是运行证据，不是正式备份。

### 10.2 执行顺序

1. 生成审阅页并通过合同测试。
2. 备份 MySQL 与 Redis，校验文件和哈希。
3. 运行 content-pack dry-run。
4. 精确清理两条已确认的旧测试评论。
5. 正式导入完整内容包。
6. 等待并确认目标文章 pending 聚合为空。
7. 核对评论、关系、bitmap、SDS、用户计数、缓存与搜索索引。
8. 以同一包再执行一次正式导入，确认数量、位图和计数不再变化。
9. 浏览器验收 Hub、文章详情、评论线程、作者主页和关注列表。

出现错误时停止后续步骤，保留本次备份和导入报告。MySQL 可从 SQL 备份恢复；Redis 可从 RDB 恢复。恢复属于显式运维动作，不在失败处理器中自动执行。

## 11. 审阅与证据

审阅生成器增加 Node 测试，覆盖：

- HTML 转义，不允许评论正文或昵称注入页面。
- 输出必须位于仓库内且被 Git 忽略的目录。
- 统计卡显示总评论、顶级/回复、点赞、收藏、关注、浏览目标和文章覆盖。
- 按账号显示评论、点赞、收藏、关注出度/入度。
- 按文章显示评论线程数和互动热度，并明确列出零评论目标。
- 显示最早/最晚评论时间及月份分布，暴露批量时间痕迹。

运行时证据保存在 `.codex-tmp/community-interactions/`，包括审阅页、dry-run 输出、备份 metadata 引用、正式导入摘要、幂等复跑摘要和核验查询结果。它们保持 Git 忽略，正式数据定义仍由仓库中的 YAML、代码和测试承担。

## 12. 验证策略

### 自动化测试

- manifest 与 Validator 数量、覆盖、时间、层级和唯一性测试。
- 反应对账服务的受管理位新增/撤销、真实用户保留、孤儿用户清理、SDS 精确重建和无事件测试。
- 外部站长关注的解析、所有权限制、重跑、停用和用户计数重建测试。
- 导入后处理顺序测试：反应事实对账必须先于用户计数、缓存和索引。
- Redis 备份脚本的路径门禁、参数、容器命令、哈希 metadata 与失败清理合同测试。
- 审阅页转义、路径和统计测试。

### 运行态核验

- 96 条有效内容包评论，72 条顶级、24 条回复；44 个目标全部有评论。
- 168 个受管理点赞位、80 个受管理收藏位；所有声明位存在，未声明的受管理位不存在。
- 目标文章不存在用户的 bitmap 位为 0；SDS like/fav 等于所有有效 bitmap 位总数。
- 28 条有效受管理关注，两侧关系镜像一致；站长收到 6 条入站关注但没有新增出站关注。
- 44 个目标浏览量不低于 manifest minimum。
- 同一内容包二次导入后上述结果不变化，也不新增通知。
- 两条旧测试评论不可见，站长文章和评论不被修改。

### 完整工程验证

- 后端定向测试与 `mvn test`。
- 必要的 Testcontainers 集成测试。
- 前端 `npm run test:run` 与 `npm run build`。
- `git diff --check`、忽略文件审计和任务范围审计。

## 13. 验收标准

- 打开任意公开文章都能看到至少一条与正文细节相关的评论；热点与冷门文章密度不同。
- 评论时间跨越文章生命周期，八个账号的说话方式和参与主题有明显差异。
- 作者主页上的关注/粉丝、文章详情点赞/收藏和评论总数与权威数据一致。
- 历史导入没有产生通知，也没有创建任何 `Rekyrice` 主动行为。
- Redis 孤儿互动位被清理，受管理反应可通过同一 manifest 精确重放。
- MySQL 和 Redis 均有可校验的仓库外备份，导入证据完整。
- 正式导入和第二次幂等导入均成功，自动化测试、构建和浏览器验收通过。
