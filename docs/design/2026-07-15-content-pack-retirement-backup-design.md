# 内容包旧内容退役与备份设计

## 1. 背景与目标

`content-v3` 已通过版本化内容包导入 40 篇文章和 8 个社区账号。当前数据库还保留 34 篇已软删除的旧初始化帖子，其中 32 篇来自八个旧 Seed 账号，2 篇来自旧兼容账号；Rekyrice 的 4 篇原有文章必须长期保留。

当前清理结果只存在于运行中的 MySQL、Elasticsearch 和审计表。若在全新数据库上重新执行旧 Seed，再导入 `content-v3`，旧帖子仍可能重新出现。本设计把这次人工判断固化为可审计、可重复、默认拒绝扩大范围的内容包退役规则，并建立内容源与运行数据的分层备份方案。

目标：

- 全新环境执行“旧 Seed → `content-v3`”后，自动得到 40 篇内容包文章，并完整保留导入前已经存在的全部 Rekyrice 文章；当前正式数据库因此保持 44 篇公开文章。
- 退役范围严格限定为 34 个明确声明的旧 slug。
- 任何站长帖子都不能被内容包退役逻辑删除。
- 重复导入不产生新增删除或重复副作用。
- 内容源、数据库运行状态和可重建索引采用不同的备份策略。

## 2. 范围

本阶段包含：

- 旧帖子退役清单、加载、校验、事务内软删除及导入报告。
- 标签计数、缓存、作者计数和搜索索引的后置同步。
- MySQL 备份脚本和备份清单。
- 不破坏当前数据库的全新数据库演练。
- 首页、文章详情、作者页和媒体资源验收。

本阶段不包含：

- 新增评论、关注、点赞、收藏或浏览量。
- 修改或删除 Rekyrice 的任何帖子。
- 把数据库导出、密钥、运行日志或 `.codex-tmp` 快照提交到 Git。
- 为当前约 13.8 MB 的内容包引入 Git LFS。

互动数据将在本阶段完成后单独设计和实施。

## 3. 内容源与运行数据备份

### 3.1 Git 作为内容源备份

`content/seed/content-v3/` 中的 YAML、40 篇 Markdown 和原始图片继续由普通 Git 跟踪。它们是内容包的可重复构建来源，可以重新上传 OSS 并重建 MySQL 中的内容包实体。Rekyrice 的运行时文章和其他真实用户数据不属于内容包，必须通过 MySQL 备份恢复。

本地分支不是远端备份。代码审核完成后，由项目维护者将任务分支合并并推送到受控远端。Coding Agent 只创建本地提交，不执行 push。

### 3.2 MySQL 作为运行状态备份

新增 PowerShell 备份入口，使用 `mysqldump --single-transaction` 导出一致性快照，并生成压缩文件和 SHA-256 校验值。备份目录必须通过参数或环境变量显式配置在 D 盘独立位置；不得默认写入 C 盘、用户主目录、仓库或 Git 跟踪目录。

备份至少覆盖：

- 用户、帖子、评论、关注和互动数据。
- `seed_content_identity` 稳定映射。
- `admin_audit_logs` 和 `outbox` 审计记录。
- 数据库迁移版本。

备份脚本不把数据库密码写入命令输出、文件名或日志。执行完成后输出备份路径、字节数、校验值和数据库名称，不输出敏感值。

### 3.3 OSS、Elasticsearch 与 Redis

内容包 OSS 对象由 Git 中的原始资源派生，本阶段通过内容包重导恢复，不再复制一份同源对象。未来真实用户上传的不可再生媒体应单独启用 OSS 版本控制或跨桶备份。

Elasticsearch 和 Redis 都是可重建状态：Elasticsearch 从 MySQL 回填，Redis 从数据库与业务事件重建，不作为主要备份对象。

`.codex-tmp` 快照只用于单次导入前后的短期恢复，不承担正式备份职责。

## 4. 退役清单

在 `content/seed/content-v3/` 增加独立的 `retirements.yml`。每条记录只声明一个已确认旧帖的 slug，并可附带技术原因。清单固定包含本次已确认的 34 个 slug，不接受正则表达式、作者范围、模糊匹配或“删除全部非站长帖子”等扩大语义。

数据模型使用不可变定义，例如：

```yaml
posts:
  - slug: seed-night-coder-1
    reason: legacy-template
  - slug: 你好
    reason: legacy-placeholder
```

加载器在旧版本内容包没有该文件时返回空清单，保持向后兼容；`content-v3` 必须存在该文件。

## 5. 安全校验

正式写入前，校验器必须拒绝：

- 空 slug、重复 slug 或超过数据库字段长度的 slug。
- 退役清单与当前内容包 40 篇文章的 slug 重叠。
- 非 `complete` 阶段内容包执行正式退役。
- 清单数量不是预期的 34 条。

数据库事务中还必须执行运行时安全检查：

- 查询只使用参数化的精确 slug 集合。
- 候选帖子 `creator_id` 等于 `site.ownerUserId` 时立即失败并回滚整个内容包数据库事务。
- 候选帖子已经是 `deleted` 时视为幂等跳过。
- 清单中不存在的 slug 不产生删除；报告记录未命中项，便于全新环境与已有环境采用同一清单。
- 内容包本次写入的 40 个稳定帖子 ID 必须从退役候选中排除。

退役规则不依赖用户昵称、可变 handle、Snowflake ID 或正则模式。

## 6. 事务与后置同步

退役发生在内容包账号和文章写入完成后、MySQL 内容包事务提交前。数据库写入器执行以下步骤：

1. 解析 34 个精确 slug。
2. 加锁读取仍为 `published` 的候选帖子及其作者和标签。
3. 执行站长排除与内容包 ID 排除断言。
4. 逐篇软删除，并对其标签调用已有的安全递减逻辑。
5. 将退役帖子 ID、作者 ID 和未命中 slug 加入写入结果。

事务提交后，导入编排器对退役结果执行：

- 失效文章详情、Feed 和相关作者缓存。
- 重建相关作者的帖子计数。
- 将退役帖子在 Elasticsearch 中标记为 `deleted`。
- 在结构化导入报告中输出退役数量、ID 和未命中项。

后置同步失败时沿用现有 `partial` 语义：MySQL 事务不回滚，失败项进入报告并允许重跑修复。第二次成功重跑的退役数量应为 0。

## 7. 全新数据库演练

演练使用现有 MySQL 容器中的独立临时数据库，不删除、不重命名、不覆盖当前 `chtholly` 数据库。临时数据库名称带任务前缀和时间戳，并在操作前验证名称只包含允许字符。

演练流程：

1. 对当前数据库生成正式压缩备份和 SHA-256 校验值。
2. 创建隔离临时数据库并执行全部迁移。
3. 执行会产生旧模板帖的历史 Seed 路径。
4. 正式导入 `content-v3`。
5. 记录导入前 Rekyrice 已发布帖子集合，断言导入后该集合完整保留；同时断言 40 个内容包稳定帖子和 8 个内容包账号存在。
6. 断言 34 个退役 slug 均不在 published 集合中。
7. 再导入一次，断言没有新增帖子且退役数量为 0。
8. 验证 Elasticsearch、标签计数和作者计数。
9. 停止临时服务；确认临时数据库名称后使用 MySQL 原生命令删除该临时数据库。

演练的最终公开文章数必须等于 40 加上演练库导入前的 Rekyrice 文章数；不假设运行时手工文章能由历史 Seed 重建。任何步骤失败都保留当前正式数据库，输出失败阶段和临时数据库名称，不自动切换应用连接。

## 8. 网站验收

使用正式数据库和现有站点执行以下验收：

- 首页 Feed 恰好返回 44 篇公开帖子，不包含 34 个退役 slug。
- 40 篇内容包文章均能打开详情页。
- Rekyrice 的 4 篇原有文章继续存在。
- 8 个内容包作者页可访问，文章数量与数据库一致。
- 抽样 Markdown、封面、正文图片和头像均返回 HTTP 200。
- Elasticsearch 的 published 数量与 MySQL 一致。

## 9. 测试策略

实现遵循测试驱动：

- Loader 测试：文件存在、旧版本缺失兼容、重复 slug。
- Validator 测试：数量、重叠、空值和阶段限制。
- DatabaseWriter 测试：精确退役、站长硬阻断、已删除幂等、未命中报告、内容包 ID 排除。
- ImportService 测试：提交后缓存、计数和搜索同步，后置失败返回 partial。
- Mapper XML 测试：只允许参数化值，不出现字符串拼接。
- 集成演练：独立数据库上验证首次导入与第二次幂等导入。

## 10. Git 与交付

设计、实施计划、代码、测试、`retirements.yml` 和备份脚本按独立职责提交。每次提交前执行新增忽略文件审计，数据库备份、日志和 `.codex-tmp` 内容不得暂存。

任务分支保留在独立 worktree 中。Coding Agent 不 push、不合并、不删除 worktree；远端发布由项目维护者完成。
