# Chtholly Hub 初始社区内容包设计

> 日期：2026-07-11
> 状态：已确认
> 适用范围：现有 Seed 账号、种子文章、媒体与社区互动的原位替换

## 1. 背景与问题

当前 Seed 系统已经创建了 8 个账号和 32 篇种子文章，加上站长及测试内容，MySQL 与 Elasticsearch 中共有 37 篇公开文章。Hub 首页每页固定显示 8 篇，因此首页只能直接看到不到 10 篇，但底部分页可以继续浏览；文章数量并未丢失。

真正需要解决的是内容质量和社区观感：

- 32 篇文章由少数通用模板展开，同类文章共享开头、段落结构和结尾，AI 痕迹明显。
- 所有种子文章的 `img_urls` 为空，卡片没有封面，正文没有叙事图片。
- 8 个头像文件名不同但内容哈希相同，实际显示同一张默认图。
- “夜猫子程序员”“番剧评论家”等昵称直接暴露职业标签，不像普通社区用户。
- 每个账号固定 4 篇、固定评论数和固定关注关系，数据分布过于整齐。
- 当前运行时 LLM 生成方式不可复现，不适合逐篇审阅、定点修改和稳定重跑。

## 2. 目标

构建一个可版本管理、可审阅、可重复导入的 `content-v2` 内容包，原位替换现有低质量 Seed 数据：

- 保留现有用户 ID、32 篇旧 Seed 文章的文章 ID，以及可以继续成立的互动关系；新增的 8 篇文章使用新 ID。
- 更新 8 个账号的公开 nickname、handle、简介、标签和头像。
- 将现有低质量种子文章替换为 40 篇策划内容；不足部分新增，现有文章优先原位更新。
- 内容比例约为动漫 50%、Java/Python 后端与 AI/Agent 25%、生活/摄影/阅读 25%。
- 每篇文章有封面，至少 60% 的文章包含 1–3 张有叙事作用的正文图片。
- 所有正文、媒体、MySQL 元数据、缓存与 Elasticsearch 索引保持一致。
- 允许重复运行；相同内容包不得生成重复用户、文章或互动数据。
- 用户先审阅 8–10 篇代表作，通过后再完成其余内容。

## 3. 非目标

- 不把 Hub 首页改成一次性展示 40 篇。首页继续每页 8 篇，通过分页浏览全部内容。
- 不建设通用 CMS 或新的 Admin 内容导入界面。
- 不在应用启动时临时调用 LLM 生成正式种子文章。
- 不生成前端、UI/UX 或与项目维护者技术方向无关的技术文章。
- 不使用真实陌生人的照片作为账号头像，也不直接使用可识别的现成动漫角色头像。
- 不为了本任务重构无关的 Post、Search、Agent 或 Storage 领域。

## 5. 内容包结构

建议新增以下结构：

```text
content/seed/content-v2/
├── manifest.yml
├── accounts.yml
├── assets.yml
├── posts/
│   ├── <post-seed-key>.md
│   └── ...
└── assets/
    ├── avatars/
    ├── covers/
    └── inline/
```

### 5.1 `manifest.yml`

记录内容包版本、目标文章数量、导入约束和内容哈希。版本使用 `content-v2`，不复用现有 `full` 或 `seed:content_only` marker。

### 5.2 `accounts.yml`

每个账号至少包含：

- 不可变 `seedKey`。
- 公开 nickname、handle、bio、gender、birthday、school 和 tags。
- avatar asset key。
- 详细写作人格：句长、常用语、兴趣、偏见、知识边界、禁用表达。
- 旧账号定位信息，用于首次迁移到新的公开 handle。

### 5.3 文章 Markdown

每篇文章以 Markdown 正文为主，元数据由 YAML front matter 或 `manifest.yml` 引用，至少包含：

- 不可变 `seedKey`。
- author seedKey。
- 公开 title、slug、description 和 tags。
- publishTime。
- cover asset key。
- inline asset keys。
- 写作 Brief：事实锚点、作者立场、结构形式、资料来源。
- 内容哈希和媒体哈希。

公开 slug 可以调整；导入器不能用公开 slug 作为唯一幂等键。

### 5.4 `assets.yml`

每个素材记录：

- asset key、用途、来源类型和原始位置。
- 本地相对路径或远程来源 URL。
- 来源说明。
- 内容 SHA-256。
- 目标格式、尺寸、裁切方式和最大体积。
- 导入后的稳定存储 object key。

## 6. 头像与媒体策略

### 6.1 头像

8 个头像采用二次元为主的混合方案：

- 5 个原创二次元人物头像，画风、色调、构图和背景必须有明显差异。
- 1 个动物或手绘吉祥物头像。
- 1 个像素画或游戏图标头像。
- 1 个风景或摄影裁切头像。

头像统一输出为 512×512 WebP，但不能使用同一角色换色、同一模板批量换背景或写实 AI 人脸。候选素材可以来自已确认可用的本地图片，也可以使用有合适授权的网络素材或原创生成素材。

### 6.2 文章图片

- 40/40 篇有封面。
- 至少 24 篇含正文图，目标为约 26 篇。
- 动漫文章使用海报、官方宣传图、合适的场景截图或氛围裁切；涉及具体集数时图片必须与正文讨论一致。
- 技术文章使用架构图、终端输出、指标图、运行结果或必要代码截图，不使用无关的“电脑桌面”图库。
- 摄影和生活文章可以由图片组织叙事；阅读文章使用书页局部、书桌或环境图。
- 图片必须有实际叙事作用，不为了满足数量强行插图。

远程图片不能作为正文长期依赖。导入阶段需要下载、校验、转码并通过现有 `StorageService` 转存到 OSS 或 local storage；最终 Markdown 和 `img_urls` 使用站点自己的稳定 URL。

## 7. 内容生产与去 AI 味流程

### 7.1 逐篇 Brief

每篇文章在写正文前必须明确：

1. 事实锚点：版本、报错、集数、时间点、地点、物件或可信资料。
2. 作者指纹：句长、语气、常用词、偏见、知识边界和不会说的话。
3. 核心立场：作者真正赞成、反对或仍不确定的内容。
4. 结构形式：排障记录、实验笔记、短评、清单、照片日志、复盘或长评。
5. 媒体计划：封面和每张正文图出现的原因。
6. 资料来源：技术官方文档、Bangumi/作品官方信息、书目信息及图片来源。

### 7.2 编辑规则

- 技术文必须有具体输入、环境、现象和结果；代码片段需经过语法或运行验证。
- 动漫文必须基于正确作品、角色、集数和镜头信息；不编造台词或剧情。
- 生活文必须包含具体时间、地点、物件、行动或感官细节，避免泛化“治愈感”。
- 不强制所有文章使用 H2/H3、开头引入和结尾总结。
- 禁止批量使用“首先、其次、最后、总之、值得一提、在这个时代、让我们”等套话。
- 不强求积极结论；允许犹豫、偏爱、反对、跑题和没有解决的问题。
- 同一作者保持稳定声音，不同作者不得共享开头、结尾和段落节奏。

### 7.3 五道质量门禁

1. 事实核验。
2. 作者声音编辑。
3. AI 套话和重复句式扫描。
4. 跨文章标题、开头、段落及 Embedding 相似度去重。
5. 8–10 篇代表作的用户抽样复核。

代表作需要覆盖动漫长评、动漫短评、Java/Python 排障、AI/Agent 实验、摄影和阅读。只有代表作通过后才继续完成余下文章。

## 8. 社区真实感

现有固定数量互动需要改成不均匀分布：

- 发布时间分散在约 90–120 天，单日 0–3 篇，不使用统一小时和固定间隔。
- 每个账号 4–6 篇，允许出现明显的活跃期和停更期。
- 单篇评论 0–6 条，允许无人评论；评论长度和语气不同，回复必须引用正文中的具体观点。
- 点赞、收藏和浏览使用长尾分布，少量热帖明显高于普通帖子，点赞与收藏不必同比例。
- 关注关系保持稀疏，不再让每个账号固定关注相同数量的人。
- 评论和关注依据共同兴趣与已建立关系生成，不使用轮转下标制造规律。
- 互动内容在文章正文确定后生成，不能只根据标题写泛化夸奖。
- 只调整 Seed 账号之间由旧 Seed 系统生成的互动；真实注册用户产生的评论、关注、点赞和收藏不得被内容包覆盖。
- 旧的泛化 Seed 评论优先按原评论 ID 原位重写；确需增删的 Seed 互动使用独立 interaction seedKey 幂等处理。

## 9. 导入流程

```text
读取 content-v2
  → dry-run 完整校验
  → 解析并转存媒体
  → 事务内按 seedKey 原位 upsert 用户、文章和互动
  → 重建标签计数
  → 失效 Feed/详情缓存
  → 同步 Elasticsearch
  → 输出验收报告
```

首次迁移时，导入器通过内部旧 seed 信息找到已有记录，建立 seedKey 到数据库 ID 的稳定映射；迁移完成后，所有后续运行只依赖 seedKey 映射和内容哈希。

导入器必须复用现有发布、缓存失效、Storage 和 Search Index 边界，避免复制一套与正式业务不同的写入语义。

## 10. 幂等、失败处理与回滚

### 10.1 幂等

- 使用 `contentPackVersion + seedKey + contentHash` 判断新增、更新或跳过。
- 同一内容包重复运行不得新增重复用户、文章、评论、关注或计数。
- 公开 nickname、handle、slug 和标题均不得充当不可变键。
- 导入报告记录每个条目的 `created`、`updated`、`unchanged` 或 `failed` 状态。

### 10.2 失败处理

- Schema、Markdown、事实元数据或媒体不完整：数据库写入前整体终止。
- 媒体下载、转码、哈希或上传失败：不更新关联文章；本次对象使用 runId 前缀，失败后清理孤儿对象。
- MySQL 写入失败：事务回滚，不执行缓存失效和 ES 同步。
- Elasticsearch 同步失败：MySQL 仍为事实源，运行标记为 partial，并支持只重试索引同步。
- 任意失败都必须记录具体 seedKey 和阶段，禁止吞掉异常。

### 10.3 回滚保护

- 正式导入前导出现有 Seed 用户和文章元数据快照。
- 旧正文和旧媒体对象在完整验收前保留，不立即删除。
- 内容包本身由 Git 版本控制，可使用上一版本重新导入。
- 不使用破坏性 SQL 批量删除现有用户或文章。

## 11. 测试与验收

### 11.1 自动测试

至少覆盖：

- manifest、account、post、asset schema 校验。
- 重复 seedKey、缺失作者、缺失素材和非法路径拒绝。
- 公开 handle/slug 改名后仍能原位更新旧 ID。
- 同一内容包运行两次不产生重复数据。
- contentHash 未变化时跳过写入。
- 媒体失败时数据库不更新。
- MySQL 成功、ES 失败时产生 partial 报告并可重试。
- 标签计数、缓存失效和 ES 索引同步。
- 图片尺寸、格式、体积和可访问性校验。

### 11.2 最终验收线

- 8 个账号使用不同头像，并拥有自然昵称、handle、简介和兴趣标签。
- MySQL 中至少有 40 篇目标公开文章；ES 中对应文章数量一致。
- 40 篇全部有封面，至少 60% 有正文图片。
- 所有正文和媒体稳定 URL 可访问。
- Hub 返回完整 total，约 5 页可以翻完全部内容。
- 标题、正文和标签可以通过搜索找到。
- 原有用户 ID、32 篇旧 Seed 文章的文章 ID，以及所有真实用户互动关系不变。
- 重复导入不会新增重复数据。
- 代表作经用户抽查通过，套话与跨文重复扫描通过。
- 后端相关测试、完整 `mvn test`、前端 `npm run build`、`git diff --check` 全部通过。

## 12. 分阶段交付

### 阶段 1：内容包基础设施

实现 manifest/schema、稳定 seedKey、dry-run、媒体处理、导入报告和单元测试。

### 阶段 2：账号与代表作

完成 8 个账号资料和头像，以及覆盖三类内容的 8–10 篇代表作。用户审阅并给出统一反馈。

### 阶段 3：完整内容与互动

按已确认反馈完成剩余 30–32 篇、封面、正文图、评论与非均匀互动数据。

### 阶段 4：真实导入与全量验收

在本地基础设施上执行 dry-run 和正式导入，核对 MySQL、Elasticsearch、Storage 和 Hub，再运行完整后端测试与前端构建。

每个阶段应独立验证并使用中文 Conventional Commit 提交；不得推送远端。

## 13. 2026-07-12 真实来源重做修订

阶段 2 的首批内容经用户审阅后未通过：文章结构、头像和配图均存在明显的 AI 生成或批量设计感。此前的账号与代表作设计由本节替代；已经实现并验证的内容包导入基础设施继续保留。

本节与前文冲突时以本节为准。第 11 节“40 篇全部有封面、至少 60% 有正文图片”的验收项取消，改为媒体来源真实、与正文直接相关即可；文章允许只有封面或完全无图。

### 13.1 版本与范围

- 被否决的 `content-v2` 不进入正式导入流程。
- 实现阶段新建 `content-v3`，清理分支中仅服务于旧草稿的账号、文章和媒体文件。
- 目标仍为 8 个账号和约 40 篇文章：动漫 20 篇，Java/Python/LLM/Agent 10 篇，游戏、读书、摄影和社区杂谈 10 篇。
- 扩写前只制作 3 篇样稿：Bangumi 争议短评、真实 GitHub Issue 技术帖、数百字社区短帖。用户明确通过后才能继续。

### 13.2 真实来源重构

每篇文章先建立来源卡，记录来源页面标题与 URL、来源类型、抓取日期、可核查事实、观点方向、图片原始地址、哈希和使用说明。正文不得沿用单篇来源的段落结构，也不得通过同义词替换制造近似副本。允许使用一小句带作者和链接的短引用，文章主体必须重新选择材料、组织论点和表达。

- 动漫内容来自 Bangumi 条目、公开日志、讨论页和动画官网，必须落到具体作品、角色、集数、演出或真实争议。
- 技术内容来自真实 GitHub Issue、官方文档、公开故障复盘和可重复样例，采用问题整理、Issue 复现或差异记录等诚实定位。
- 杂谈内容来自真实讨论主题、书籍资料、游戏公告或公开摄影作品，不虚构生活经历。
- 禁止批量使用“不是……而是……”“真正让我在意的是……”等模板句；不要求完整的开场、转折、升华和总结；允许短句、省略、偏题和未完成结论。

### 13.3 账号与头像

现有 8 个账号的昵称、人设、简介和头像全部替换。新账号不得复制任何真实 Bangumi 用户的昵称、简介、收藏或文章。昵称采用长短不一的论坛风格；简介最多一句，也允许留空。

头像可以从 Bangumi 公开用户资料 API 获取，但仅复用公开图片，不关联原用户身份。最终比例为 6 个动漫或游戏相关头像、1 个动物头像、1 个风景或普通杂图头像；不使用女生动漫头像，不使用 AI 生成头像，也不从全身角色立绘硬裁头像。当前确认的视觉候选为 C01、C03、C04、C05、C09、C12、仓鼠 C06 和夜空 C08。

### 13.4 媒体规则

- 动漫封面使用 Bangumi 条目图、动画官网视觉图或公开剧照。
- 正文图使用具体集数剧照、官网截图或与论点直接相关的页面局部。
- 技术文章使用真实终端输出、代码差异、Issue 截图或文档截图。
- 取消 AI 插画、AI 风景图和装饰性 SVG 信息图。
- 不强制每篇都有正文图；允许只有封面或完全无图。
- 所有网络图片下载后固定在内容包内，记录原始地址、来源页面、抓取日期、SHA-256、尺寸和内容类型；导入过程不得依赖外部网络。

用户接受使用公开网络图片并记录出处，即使其商业转载授权不明确；来源卡仍必须保留使用说明，便于后续替换。

自动校验必须覆盖：来源字段完整、禁止 AI 生成来源、禁止被否决的旧资产进入 `content-v3`、图片文件与哈希一致、Markdown 媒体引用可解析、无外网时 dry-run 仍能完成。
